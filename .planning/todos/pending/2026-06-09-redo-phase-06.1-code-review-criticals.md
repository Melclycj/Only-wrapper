## Problem

The phase 06.1 code-review remediation (CR-01..CR-04 + WR-02) was **REVERTED** on
2026-06-09 (commits `fda9fe3`, `d765fa9` revert `4ae8e1a` + `50d6c41`). Storage
(`session-store.ts` `flush()`) is restored to the user-approved-good version.

**Cautionary precedent — why it was reverted:** the remediation was rushed. Its
CR-03 fix ("coalesce" the in-flight write) **DROPPED mid-write mutations** — a
mutation that landed while a write was in flight could be silently lost, leaving
stale data on disk. That is a *worse* data-loss bug than the one CR-03 set out to
fix. The automated suite went GREEN while this (and the duplicate-Start UI defect)
was broken, so **green is not proof** — the user must re-verify in the running app.

The CR/WR findings themselves are still REAL (see `06.1-REVIEW.md`) and remain
unaddressed after the revert. They must be redone — but carefully, with tests that
exercise the REAL code path (not stubs), and a MANDATORY user re-verify afterward.

## Findings to redo (all currently UNADDRESSED after the revert)

- **CR-01 — dock-relaunch quit-flag.** The `quitting` flag never resets, so after a
  survived quit cycle (macOS dock relaunch / `activate`) the before-quit flush is
  skipped → the trailing debounced write can be lost. Fix: reset the flag on
  `activate`/relaunch. (`src/main/index.ts`)

- **CR-02 — concurrent-restart lock.** `restart()` has no in-flight lock → a rapid
  double Restart can double-spawn. NOTE the latent edge case the reverted fix
  introduced: a FAILED respawn could leave the session stuck in `restarting`
  (the lock never cleared on the error path). The redo MUST clear the lock in a
  `finally` so a failed respawn cannot wedge the session. Add a test that drives a
  failing respawn, not just the happy path. (`src/main/pty-manager.ts`)

- **CR-03 — failed-write data loss (the ORIGINAL real concern).** `flush()` clears
  `dirty` before the write resolves; if the write FAILS, the change is lost with the
  flag already cleared. The rushed "coalesce" fix DROPPED mid-write mutations and was
  reverted. **Correct fix:** clear `dirty` only AFTER the write RESOLVES, and if
  `dirty` became true again DURING the write (a mutation arrived in flight), issue a
  FOLLOW-UP write — never coalesce away the in-flight mutation. The restored flush()
  + the new R2 regression guard (`session-store.test.ts`, commit `6257097`) lock the
  "no-drop" half of this contract; the redo must add the "write-failure does not
  clear dirty" half. (`src/main/session-store.ts`)

- **CR-04 — setOrder clamp.** `setOrder()` casts an unchecked IPC string to
  `LogicalId` and accepts an unclamped order → can poison `nextOrder()` /
  map-poisoning (defense-in-depth). Fix: validate the id + clamp order to a bounded
  range. (`src/main/pty-manager.ts`)

- **WR-02 — handleRestart pid>0 guard.** `handleRestart` optimistically writes
  `status: 'running'` before `onPtyStatus` confirms, with no `pid > 0` guard (unlike
  `handleStart`, which already guards). A failed respawn (pid -1) would clobber the
  error card with a false 'running'. Fix: mirror `handleStart`'s `pid > 0` guard (a
  shared `optimisticSpawnFlip` helper is the natural shape). (`src/renderer/SessionManager.tsx`)

## Solution

Redo each finding with care, following TDD against the REAL path:

1. Write a RED test that exercises the REAL code path (not a stub) and fails for the
   right reason. For CR-02/CR-03 include the failure/edge path (failed respawn;
   write rejection), not only the happy path.
2. Implement the minimal fix; GREEN.
3. Run the full gate: `tsc`, `eslint`, `test:unit`, `package` + `test:smoke`.
4. **MANDATORY user re-verify in the running app** of start / restart / persistence
   (quit→relaunch round-trip; rapid double-restart; mid-write mutation durability).
   Automated green is NOT sufficient — the suite passed while the reverted remediation
   was actively broken. The user signs off on the live app before the phase advances.
