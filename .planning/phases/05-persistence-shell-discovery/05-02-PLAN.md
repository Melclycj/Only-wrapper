<objective>
The persistence vertical slice. After this plan, a user can close the app fully, reopen it,
and see every session profile restored â€” dormant (`not_started`), in saved order â€” with the
store written through lowdb behind a debounced + quit-flushed write. This is the end-to-end
PERS-01/PERS-02 capability; the UI affordances to START a dormant session and the empty/idle
cards land in Plan 05-03, but the data round-trip is real and observable here (the renderer's
existing reconcile/listSessions path already renders restored rows).

Three pieces:
1. **SessionStore** (`src/main/session-store.ts`) â€” owns a lowdb `Low<StoreSchema>` loaded via
   the load-bearing dynamic `import('lowdb')` (Pitfall 1), coerces every record on load
   (D-01), backs up + recovers from a corrupt file (never crashes), and exposes
   `scheduleSave()` (debounced ~300ms) + `flush()` (quit) per D-13.
2. **PtyManager.hydrate / store-backed setters** â€” `hydrate(records)` populates a SEPARATE
   `dormantRecords` map (Pattern 4 option b â€” preserves the "every PtySession has a live pty"
   invariant); `listSessions()` merges live + dormant; `create({id})` promotes a dormant id
   into a live PtySession. The validate-in-main `setOrder`/`setUiState` from Plan 05-01 now
   call `store.scheduleSave()`, and every record mutation signals the store.
3. **index.ts lifecycle wiring** â€” `whenReady` does `store.load() â†’ ptyManager.hydrate(...) â†’
   createWindow()` restoring validated window bounds (D-12) BEFORE show; `before-quit` adds
   the `preventDefault()`-then-flush re-entrancy guard so the trailing debounced write lands.

Purpose: deliver the durable, dormant-restore data layer that the whole phase is named for,
with the one genuinely novel runtime surface (lowdb ESM-in-CJS) proven in the BUILT app.

Output: SessionStore, hydrate + store-backed setters, lifecycle wiring, GREEN unit + smoke.
</objective>
