## Summary

Reviewed the TERM-05 startup-command auto-run implementation: the readiness-probe
seam (`readiness-probe.ts`), the probe-then-inject state machine inside
`PtyManager.create()`, the additive `notice` IPC field, and the renderer's
notice rendering. The probe-invisibility lever (D-02) and the no-secret-logging
posture (V7) are largely sound. However, two correctness defects break the D-04
fallback UX and the restart auto-run path, and several robustness gaps around
listener lifecycle and edge-case matching warrant fixing.

Most serious:
- **CR-01** — the timeout ready-fail notice rides a `status: 'running'` event,
  which makes the renderer emit a spurious "— restarted —" separator (false UX).
- **CR-02** — `restart()` registers a one-shot `onExit` that is never disposed
  and double-fires the respawn under the probe-then-inject path, and more
  generally stacks listeners across restarts.

## Critical Issues

### CR-01: Timeout notice triggers a spurious "— restarted —" separator in the renderer

**File:** `src/renderer/SessionView.tsx:215-232`, paired with `src/main/pty-manager.ts:410-415`

**Issue:** On readiness-probe timeout, main emits the ready-fail notice over the
`pty:status` channel carrying the *current* lifecycle status, which is `'running'`
(pty-manager.ts:410-414: `liveStatus = this.sessions.get(id)?.status ?? 'running'`).

In the renderer's `onPtyStatus` handler, the status branch runs *before* the
notice branch:

```tsx
if (p.status === 'running') {
  if (hasRunBeforeRef.current) {                // already TRUE after the spawn
    term.write(`\r\n\x1b[2m— restarted ${hhmm} —\x1b[0m\r\n`);  // SPURIOUS
  }
  hasRunBeforeRef.current = true;
}
if (p.notice) { term.write(`\r\n\x1b[2m— ${p.notice} —\x1b[0m\r\n`); }
```

The spawn already fired `setStatus(id, 'running')` (pty-manager.ts:314), so
`hasRunBeforeRef.current` is `true` by the time the 4s timeout notice arrives.
The handler therefore writes a fake `— restarted HH:MM —` line immediately above
the ready-fail notice, falsely telling the user the session restarted. This
directly violates the D-04 intent that the notice be *additive and must not
regress/confuse the lifecycle UI*.

**Fix:** Make the notice path mutually exclusive from the running-transition path
in the renderer (a notice-bearing event is not a lifecycle transition), e.g.:

```tsx
const offStatus = window.api.onPtyStatus(id, (p) => {
  if (p.notice) {
    term.write(`\r\n\x1b[2m— ${p.notice} —\x1b[0m\r\n`);
    return; // a notice event is informational only — not a 'running' transition
  }
  if (p.status === 'running') {
    if (hasRunBeforeRef.current) {
      const hhmm = new Date().toTimeString().slice(0, 5);
      term.write(`\r\n\x1b[2m— restarted ${hhmm} —\x1b[0m\r\n`);
    }
    hasRunBeforeRef.current = true;
  }
});
```

Alternatively, main should send the notice on a status value that the renderer
does not treat as a restart trigger, but the renderer-side guard is the robust fix.

### CR-02: `restart()` leaks/duplicates an `onExit` listener; under auto-run it can respawn twice

**File:** `src/main/pty-manager.ts:613-616` (with `create()`'s own `onExit` at 421)

**Issue:** `restart()` attaches a one-shot respawn via `s.pty.onExit(() => respawn())`
(line 615) but never disposes the returned disposable. `create()` also installs
its own permanent `onExit` (line 421). node-pty fires `onExit` to *all* registered
listeners, so:

1. The restart-added listener is never cleaned up. Because the same `IPty`
   instance can have multiple `onExit` callbacks accumulate, every restart adds a
   new respawn closure bound to a stale `record`. While the old pty is replaced on
   respawn, the listener disposable being dropped is a latent leak and an
   ordering hazard.
2. More concretely: `respawn()` calls `create()` which, for a session whose
   `startupCommand` is non-empty, starts the probe state machine and `setTimeout`.
   `restart()` resolves its promise from inside `respawn()`. There is no guard
   that `respawn()` runs exactly once — if `onExit` is ever delivered more than
   once by node-pty (or if `stop()`'s SIGTERM→SIGKILL path produces an exit that
   races), `respawn()` (and thus a second `create()` + second probe + second
   injected command) can fire twice for one restart, violating the "one live pty
   per id" invariant and SC1's inject-ONCE guarantee.

**Fix:** Capture and dispose the one-shot restart listener, and guard the respawn
against double invocation:

```ts
if (s.alive) {
  let respawned = false;
  const off = s.pty.onExit(() => {
    if (respawned) return;
    respawned = true;
    off.dispose();
    respawn();
  });
  this.stop(id);
} else {
  respawn();
}
```

## Warnings

### WR-01: `wireNormalOnData` stacks a second `onData` listener on the same child

**File:** `src/main/pty-manager.ts:353-385, 463-467`

**Issue:** On the non-empty-command path, `create()` registers the probe listener
via `child.onData(...)` (line 353) and stores its disposable as `offProbe`. On
match it calls `offProbe.dispose()` then `wireNormalOnData(id, child)` which calls
`child.onData(...)` again — fine. But on the **timeout** path (line 397-418) it
does NOT call `offProbe.dispose()` before `wireNormalOnData` — wait, it does call
`offProbe.dispose()` at line 400. Good. However the post-settle scrub branch
(lines 356-366) keeps the *probe* listener alive and forwarding scrubbed bytes,
while `wireNormalOnData` has ALSO registered a second listener after match. The
result: after a successful match, BOTH the probe listener (still subscribed,
running the `settled` scrub branch) AND the normal-forwarding listener are
attached to the child, because `offProbe.dispose()` is called on match (line 374)
— so this is actually OK on the match path. Re-verify: on match, `offProbe.dispose()`
runs, so the scrub branch in the probe listener can never run afterward — meaning
the D-02 post-settle scrub (lines 356-366) is **dead code on the match path**
(the listener that contains it is disposed at the moment `settled` becomes true).

**Fix:** Either (a) do NOT dispose `offProbe` on match and instead let the
`settled` branch handle scrubbing (then do not call `wireNormalOnData`), or
(b) accept that scrubbing is handled by `stripProbeEcho` only in a path that is
currently unreachable and remove the scrub branch to avoid confusion. As written,
the adversarial-chunk scrub the comment promises (Open Q3) never executes because
the listener holding it is disposed before any post-settle byte can arrive.

### WR-02: Probe matcher can false-positive on a single chunk that contains the marker echo followed by any newline

**File:** `src/main/readiness-probe.ts:71`

**Issue:** `re = new RegExp(`${safe}[^\\n]*\\n[\\s\\S]*`)`. This matches the nonce
followed (on the same line) by any chars then a newline. The shell's *echo* of the
typed marker line `: <nonce>\r\n` itself satisfies this when the TTY echoes input
with a trailing newline before the shell has actually processed/re-prompted — the
intended send-vs-match split (Pitfall 1) is only guaranteed if the echoed input
line is NOT terminated by `\n` in the same chunk. Under line-buffered echo or
chunk coalescing, `: <nonce>\n` (echo) alone matches, defeating the
"processed-not-merely-echoed" guarantee and injecting the command before the
shell is genuinely ready.

**Fix:** Require the nonce to appear on a line that is NOT the echo line — e.g.
match the nonce occurring a *second* time, or require a prompt-shaped suffix, or
match `\n` + nonce (nonce at start of a produced line). Tune against real cold
zsh/bash captures (the file already flags this as E2E-tunable, but the current
shape is demonstrably echo-trippable).

### WR-03: Unbounded probe buffer growth before match/timeout

**File:** `src/main/pty-manager.ts:368`

**Issue:** `buffer += data` accumulates every pre-match byte for up to
`READINESS_TIMEOUT_MS` (4s) with no cap. A shell (or adversarial program that
somehow starts before the prompt) emitting high-volume output during the probe
window grows `buffer` without bound, and the entire blob is then flushed at once
on timeout. While performance is out of scope, this is also a correctness/robustness
concern: the matcher runs `re.test(buffer)` on an ever-growing string each chunk.

**Fix:** Cap the buffered bytes (e.g. keep only the last N KB, enough to contain
the marker + a prompt) and run `matches()` against the bounded tail.

### WR-04: `notice` field rides `pty:status` with no schema/length guard at the renderer

**File:** `src/renderer/SessionView.tsx:230-231`, `src/preload/index.ts:114-121`

**Issue:** The renderer writes `p.notice` straight into the terminal with ANSI
wrapper sequences. Today main only ever sends the fixed `READINESS_FAIL_NOTICE`
literal, so this is safe in practice. But the renderer trusts whatever arrives on
the channel; the preload forwards the payload verbatim. If any future/forged main
path sets `notice` to attacker-influenced content, raw control sequences would be
written to the terminal. Defense-in-depth: the renderer should treat `notice` as
display-only text.

**Fix:** Sanitize/strip control characters from `p.notice` before `term.write`,
or constrain to a known set of notice codes rather than free-form strings.

### WR-05: Whitespace-only startup command is trimmed for the gate but the stored value is preserved untrimmed

**File:** `src/main/pty-manager.ts:338-339`

**Issue:** `const cmd = record.startupCommand?.trim();` correctly skips the probe
for whitespace-only commands. But the un-trimmed `startupCommand` remains on the
record. On a later restart the same trim happens, so behavior is consistent — yet
a command like `"  echo hi  "` is injected as `record.startupCommand?.trim()`?
No: line 383 injects `cmd + '\r'` where `cmd` IS the trimmed value, so leading/
trailing whitespace the user intentionally typed is silently dropped at inject
time but preserved in the stored profile. This split (store untrimmed, inject
trimmed) is a latent inconsistency.

**Fix:** Decide one canonical form — either trim at persist time in
`updateProfile` (so the stored value matches what is injected) or inject the
untrimmed value after the non-empty gate. Document the chosen semantics.

## Info

### IN-01: `stripProbeEcho` is effectively unreachable given CR/WR-01

**File:** `src/main/pty-manager.ts:124-130, 356-366`

**Issue:** As noted in WR-01, the only caller of `stripProbeEcho` is the
post-settle scrub branch inside the probe listener, which is disposed on match.
The helper is unit-tested directly (readiness-probe.test.ts:161-177) so it is
correct in isolation, but it guards a code path that does not execute in
production. Confirm the intended invisibility hardening actually runs.

### IN-02: `void shellPath` parameters are unused in both real and stub probes

**File:** `src/main/readiness-probe.ts:88, 102`

**Issue:** `forShell(shellPath)` ignores its argument on macOS (zsh/bash share the
probe) and on the Windows stub (throws). The `void shellPath` discard is a clear
convention, but the interface implies per-shell behavior that does not yet exist.
This is acceptable for the seam but worth a note for Phase 8 implementers.

### IN-03: Smoke test asserts ordering via `lastIndexOf`/`indexOf` which can be brittle across wraps

**File:** `tests/smoke/startup-command.smoke.test.ts:115`

**Issue:** `b.lastIndexOf('JW_STARTUP_OK') > b.indexOf('restarted')` proves the
command output appears after the restart separator, but if scrollback wraps or
the separator text 'restarted' appears in unrelated output, the assertion could
mis-evaluate. Low risk given the controlled fixture; consider anchoring on the
full `— restarted` separator string.
