# Phase 05.1: TERM-05 startup-command auto-run - Research

**Researched:** 2026-06-06
**Domain:** PTY shell-readiness detection + invisible probe + command injection (node-pty / zsh / bash on macOS, Electron main process)
**Confidence:** HIGH (architecture + injection mechanics), MEDIUM (exact probe sequence cross-shell behavior — must be E2E-verified on a real cold spawn this phase)

## Summary

This phase un-defers TERM-05: auto-run a session's saved `startupCommand` into a freshly spawned PTY once the shell is genuinely ready to accept a line of input. The phase exists because the naive fixed settle-delay approach lost/garbled keystrokes on cold first spawn. The entire phase rides on one hard problem: **reliably knowing a just-spawned interactive shell's line editor (zsh `zle` / bash `readline`) is up and listening, without persistent shell hooks and without any visible artifact.**

The single most important architectural fact discovered in the codebase: PTY output flows `child.onData → PtyManager.send('pty:data') → renderer term.write`. **A probe is invisible by construction if main intercepts the probe's echo/output inside the `onData` handler and does NOT forward those bytes to the renderer.** This is far more robust than trying to find a control sequence xterm.js will swallow — main owns the byte stream and can simply withhold the probe round-trip from the renderer until the real command is written. This is the recommended invisibility mechanism (D-02).

**Primary recommendation:** In `PtyManager.create()`, after `pty.spawn()` and only when `record.startupCommand` is non-empty, run a **one-shot output-gated readiness probe**: install a transient `onData` interceptor that (a) buffers (does NOT forward) all bytes, (b) writes a unique no-side-effect marker probe to the PTY, (c) waits until that marker's echo round-trips back, (d) on match, stops intercepting, discards the buffered probe bytes, then writes `startupCommand + "\r"` as normal typed input (which the shell echoes visibly and adds to history). On timeout (recommend ~4 s), do NOT inject — flush the buffered real shell output to the renderer (so the bare prompt appears normally), restore normal forwarding, and emit a non-intrusive notice (D-04). Put the per-shell probe knowledge behind a `ReadinessProbe` seam mirroring `ShellDiscovery`; implement macOS (zsh/bash) now, stub Windows for Phase 8 (D-03).

### Pattern 1: Output-gated one-shot readiness probe (THE crux — D-01/D-02)
**What:** After spawn, before any real output reaches the renderer, send a unique nonce marker to the PTY and wait to observe it echo back. The shell echoing the marker proves its line discipline + line editor (zle/readline) are live and reading — the exact thing the fixed delay could only guess at.

**Why a round-trip beats a delay (the race, explained):** On cold first spawn the shell process exists but its interactive read loop is not yet attached to the PTY — login-shell rc sourcing (`-l`: `.zprofile`/`.zshrc`), zle/readline init, and prompt emission all happen *after* `pty.spawn()` returns. Bytes written into the PTY during this window are delivered to the TTY but may be consumed before canonical-mode line editing is initialized, or interleaved with rc output → lost/garbled keystrokes. A byte the shell *echoes back* can only have been processed by an initialized read/echo path, so observing the echo is direct proof of readiness — not a heuristic. `[CITED: pexpect prompt-sync model — pexpect.readthedocs.io/.../replwrap.html]` `[ASSUMED: the precise rc-timing description for zsh/bash login shells — verify via E2E on a real cold spawn]`

**Marker design (avoid matching your own send — pexpect's key lesson):** The string you *send* must differ from the string you *match*, or you will match the command's own echo and declare ready one beat too early. pexpect solves this by backslash-splitting (`PS1='[PEXPECT]\$ '` sent, `\[PEXPECT\][\$\#]` matched). Two D-01-compliant adaptations that change NO shell state:

- **Option A (recommended): a no-op command whose output is a unique nonce, matched on the nonce in the OUTPUT line (not the echoed command line).** E.g. write `: READY_<uuid>\r` (`:` is the POSIX no-op builtin; the token is its argument, so nothing executes and nothing persists to env). Then match the nonce appearing on a line that is NOT the echoed input. Cleaner: `printf '<uuid>\\n'\r` — but `printf` shows the nonce in scrollback if not withheld, which is fine because main withholds ALL probe bytes anyway. **Because main discards every probe byte, even a fully-visible marker command is invisible to the user** — this is the architectural lever that makes D-02 trivial.
- **Option B: send a bare unique token followed by `\r` and rely on echo only** — simpler but riskier, since the echo appears on the same line you sent; harder to distinguish "echoed" from "processed." Prefer A.

**Invisibility mechanism (D-02) — the load-bearing insight:** Do NOT try to find a sequence xterm.js swallows. Instead, in `create()`, the probe phase installs a temporary `onData` handler that **buffers bytes and never calls `this.send()`**. When ready is detected, the probe's bytes (the marker command echo + nonce output + the fresh prompt the marker produced) are simply discarded; main then writes the real command, whose echo IS forwarded normally. The user sees only their command at a clean prompt. The renderer literally never receives the probe bytes — invisibility is guaranteed at the source, not at the display.

**Caveat — withholding the first prompt:** The probe will also consume the shell's *first* prompt render (since it ran before the marker). After injecting the real command, the shell re-renders its prompt + the echoed command. Net visible result is a clean prompt with the command typed on it — which is exactly the desired SC1 appearance. **This must be E2E-verified**: confirm no duplicate/blank prompt artifact remains. `[ASSUMED — verify by E2E]`

**Example (illustrative — sequence must be E2E-confirmed):**
```typescript
// Source: derived from pexpect REPLWrapper prompt-sync + this codebase's onData architecture
// In PtyManager.create(), after child = pty.spawn(...) and the record is built:
const probe = selectReadinessProbe(process.platform).forShell(record.shell);
const cmd = record.startupCommand?.trim();
if (cmd) {
  let buffer = '';
  let settled = false;
  const onProbeData = child.onData((data) => {
    if (settled) { this.send(PTY_CHANNELS.data, { id, data }); return; }
    buffer += data;
    if (probe.matches(buffer)) {           // marker echo observed → shell is live
      settled = true;
      clearTimeout(timer);
      onProbeData.dispose();               // stop intercepting
      // probe bytes in `buffer` are DISCARDED (invisible — D-02)
      this.wireNormalOnData(id, child);    // restore onData→send forwarding
      child.write(cmd + '\r');             // VISIBLE typed input → echoes + histfile (SC1)
    }
  });
  child.write(probe.marker);               // send the no-side-effect nonce probe
  const timer = setTimeout(() => {
    if (settled) return;
    settled = true;
    onProbeData.dispose();
    this.send(PTY_CHANNELS.data, { id, data: buffer }); // FLUSH real output → bare prompt (SC4)
    this.wireNormalOnData(id, child);
    this.send(PTY_CHANNELS.readyFail, { id });          // D-04 notice
  }, READINESS_TIMEOUT_MS);
} else {
  this.wireNormalOnData(id, child);        // empty command → normal bare shell (SC2/TERM-03)
}
```
The existing `child.onData((data) => this.send(...))` wiring (line ~281) is refactored into `wireNormalOnData()` so both the normal path and the post-probe path share one definition.

### Pattern 2: Injection that lands in history (SC1)
**What:** Write `command + "\r"` directly to the PTY in main via `child.write()`.

**Details verified against the codebase + shell behavior:**
- **Line terminator:** Use `"\r"` (carriage return, 0x0D), NOT `"\n"`. A real terminal sends CR when Enter is pressed; the TTY line discipline (ICRNL) maps CR→NL so the shell sees a completed line. The existing E2E (`pty-roundtrip.smoke.test.ts`) types text then `browser.keys(['Enter'])` which produces CR — match that. `[VERIFIED: codebase E2E + POSIX termios ICRNL behavior]`
- **History (SC1):** A command written to an interactive zsh/bash via the PTY and terminated with CR is processed exactly as typed input — it is added to the in-memory history list and (on shell exit / per histfile settings) to `.zsh_history`/`.bash_history`. No special handling needed; injection through the normal input path inherits history for free. `[ASSUMED — verify by E2E: run a command, then `history` or up-arrow shows it]`
- **Bracketed paste — do NOT use it here.** Phase 2's renderer paste path uses `term.paste()` which wraps text in bracketed-paste markers (DECSET 2004) specifically so multi-line pastes do NOT auto-execute. Auto-run is the OPPOSITE intent: it must execute. Write raw `cmd + "\r"` in main — bypass the renderer paste path entirely. The injection is a main-side `child.write`, never `term.paste`. `[VERIFIED: codebase TerminalPane.tsx lines 110-126 + CONTEXT injection-in-main lock]`
- **Trim:** Empty/whitespace-only `startupCommand` → no injection (SC2 / locked). Trim before the non-empty check.

**Anti-pattern:** Writing the command via the renderer (`window.api.ptyWrite`). CONTEXT locks injection in main; the renderer must have no auto-run write path (mirrors the IdleCard grep-enforced no-ptyWrite rule).

### Pattern 3: Single-hook trigger in create() (D-05)
**What:** All three trigger surfaces (new-with-command, Restart, dormant Start ▶ promotion) converge on `PtyManager.create()`. One probe-then-inject hook placed at the existing `child.onData` wiring point (line ~281) covers all three uniformly. Gate on `record.startupCommand?.trim()` being non-empty.

**Interaction with the `— restarted HH:MM —` separator (Phase 3, SC3):** The separator is written on the restart path. The auto-run command must land AFTER the separator. Verify ordering: if the separator is emitted to the renderer before/around respawn, the probe (which withholds bytes) must not swallow it. Recommended: emit the separator via `send('pty:data')` only after the probe settles, OR confirm the separator is a renderer-side write not subject to the probe buffer. **This ordering is a concrete plan-time decision point.** `[ASSUMED — locate the separator write and sequence it explicitly]`

**Interaction with flow-control (SC5):** The probe phase is sub-second and produces tiny output; the watermark pause/resume logic (renderer-side) is unaffected because probe bytes never reach the renderer. After the probe, normal forwarding resumes and flow-control operates unchanged. No conflict.

### Anti-Patterns to Avoid
- **Fixed settle-delay before injecting** — the original cold-spawn garble bug. The whole phase replaces this.
- **Best-effort inject after timeout** — explicitly REJECTED by D-04; it IS the garble failure mode.
- **Persistent PS1/PROMPT_COMMAND/precmd sentinel** — forbidden by D-01 (changes shell env, leaves a hook). The probe must be a one-shot that restores nothing because it changed nothing.
- **Relying on xterm to "swallow" an invisible sequence** — fragile; use main-side byte withholding instead.
- **Matching the probe's own echoed send string** — declares ready too early (pexpect's documented pitfall). Match output distinct from the send.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Knowing the shell is ready | A tuned `setTimeout` delay | The output-gated round-trip probe (Pattern 1) | Delays are machine/load-dependent and caused the original failure; a round-trip is direct proof |
| Distinguishing probe output from real output | Regex over the whole merged stream | Buffer probe bytes separately in main and never forward them | Avoids the "did the user's command produce that or did my probe?" ambiguity entirely |
| Sending Enter | Synthesizing key events through the renderer | Main-side `child.write(cmd + "\r")` | Injection is locked to main; CR is what a real Enter produces |

**Key insight:** The hard part is not the mechanism — it's *proof of readiness*. The codebase's main-owns-the-stream architecture turns "make the probe invisible" from a hard terminal-emulator problem into a trivial "don't call `send()` for these bytes" problem. Lean on that.

## Common Pitfalls

### Pitfall 1: Marking ready on the echoed command instead of the produced output
**What goes wrong:** The shell echoes the probe command bytes back immediately (line discipline echo) before it has finished processing the line; matching that echo declares ready too soon → the real command can still be garbled.
**Why it happens:** Echo of input and execution of input are different stages.
**How to avoid:** Match a nonce that appears in the command's *output* (or the prompt that follows it), distinct from the echoed input — pexpect's send-vs-match split.
**Warning signs:** Intermittent garble on slow/cold spawns even with the probe in place.

### Pitfall 2: Probe swallows the restart separator or the real prompt permanently
**What goes wrong:** Because the probe withholds bytes, the `— restarted HH:MM —` separator or the legitimate first prompt could be discarded and never shown.
**Why it happens:** Over-broad buffering / discarding.
**How to avoid:** Discard ONLY up to and including the marker round-trip; flush anything after. On timeout, flush the entire buffer (so the bare prompt shows — SC4). Sequence the separator emission relative to the probe explicitly.
**Warning signs:** Missing prompt or missing restart separator after auto-run.

### Pitfall 3: Probe never settles and blocks forever
**What goes wrong:** Marker never echoes (unusual shell, error) → terminal stuck with no output (bytes withheld indefinitely).
**Why it happens:** No timeout, or timeout path doesn't flush the buffer.
**How to avoid:** Hard timeout (~4 s recommended), and on timeout FLUSH the buffer + restore forwarding + emit notice (D-04). Never leave bytes withheld.
**Warning signs:** A started session shows a blank terminal.

### Pitfall 4: `\n` instead of `\r` produces a hung or mis-parsed line
**What goes wrong:** Writing `"\n"` may not be treated as Enter by the line editor depending on termios mapping.
**How to avoid:** Use `"\r"` (matches a real Enter / the existing E2E).
**Warning signs:** Command appears typed but does not execute.

### Pitfall 5: Bridge surface drift (security guard goes RED)
**What goes wrong:** Adding a `readyFail` event channel without updating `EXPECTED_API_KEYS` + `security.guard.test.ts` in lockstep breaks the surface assertion.
**How to avoid:** If a NEW renderer signal is needed, extend `ElectronAPI`, `preload/index.ts`, `EXPECTED_API_KEYS` (window-config.ts), and the guard test together (established 18→19 key pattern). **Lighter option: reuse the existing `pty:status` channel** — emit a status payload variant, or a transient `error`/notice via the already-exposed `onPtyStatus` subscription, avoiding a new key entirely. Recommend evaluating the reuse path first (see Open Question 1).
**Warning signs:** `security.guard.test.ts` fails `Object.keys(exposed) === EXPECTED_API_KEYS`.

## Code Examples

### Readiness-probe seam (mirrors shell-discovery.ts ShellDiscovery — D-03)
```typescript
// Source: pattern mirrored from src/main/shell-discovery.ts (this codebase)
// src/main/readiness-probe.ts — electron-free, Vitest-importable pure helpers.

/** Per-spawn readiness probe: a one-shot marker + a matcher. Changes NO shell state (D-01). */
export interface ShellReadinessProbe {
  /** Bytes to write to the PTY to elicit a detectable round-trip (no env/rc changes). */
  readonly marker: string;
  /** True once `buffer` contains evidence the shell processed the marker (not just echoed it). */
  matches(buffer: string): boolean;
}

/** The platform-aware seam (D-03), mirroring ShellDiscovery. One provider per platform. */
export interface ReadinessProbeProvider {
  forShell(shellPath: string): ShellReadinessProbe;
}

/** PURE — build a POSIX no-op probe for a unique nonce (unit-tested with a fixed nonce). */
export function buildPosixProbe(nonce: string): ShellReadinessProbe {
  // ':' is the POSIX no-op builtin; the nonce is its argument → nothing runs, nothing persists.
  // Sent string differs from the matched token to avoid matching the bare echo (pexpect lesson).
  return {
    marker: `: ${nonce}\r`,
    // Ready when the nonce appears AFTER a newline (i.e. the shell processed the line and
    // re-prompted), not merely on the echoed input line. Exact regex to be E2E-tuned.
    matches: (buf) => new RegExp(`${nonce}[^\\n]*\\n[\\s\\S]*`).test(buf),
  };
}

export class MacReadinessProbe implements ReadinessProbeProvider {
  forShell(_shellPath: string): ShellReadinessProbe {
    // zsh and bash share the POSIX ':' builtin + CR semantics; one probe covers both on macOS.
    return buildPosixProbe(`__JW_READY_${cryptoRandomHex()}__`);
  }
}

/** Windows STUB (D-03) — real PowerShell/CMD/Git Bash/WSL probes land at Phase 8. */
export class WindowsReadinessProbe implements ReadinessProbeProvider {
  forShell(_shellPath: string): ShellReadinessProbe {
    throw new Error('Windows readiness probe is implemented in Phase 8 (D-03 seam stub).');
  }
}

export function selectReadinessProbe(platform: NodeJS.Platform): ReadinessProbeProvider {
  return platform === 'win32' ? new WindowsReadinessProbe() : new MacReadinessProbe();
}
```
> The exact `marker` and `matches` regex are the researcher's discretion per D-01/D-02 and **MUST be E2E-tuned on a real cold zsh and bash spawn** — the shape above is the contract, not the final tuned sequence. `[ASSUMED — E2E-verify the regex against real zsh + bash output]`

## Open Questions

1. **New IPC key vs. reuse `pty:status` for the D-04 ready-fail notice.**
   - What we know: The bridge is exactly 18 keys, guarded by `security.guard.test.ts`. `onPtyStatus` already streams main→renderer events filtered by id.
   - What's unclear: Whether the notice fits a `pty:status` payload variant (no new key) or warrants a dedicated `onReadyFail` key (19th, lockstep update).
   - Recommendation: Prefer reusing `onPtyStatus` with a transient notice field if the renderer can render it without conflating with lifecycle status; otherwise add ONE key in lockstep. Decide at plan time; keep it minimal.

2. **Timeout duration.**
   - What we know: D-04 leaves it to Claude's discretion; "low single-digit seconds" suggested in CONTEXT.
   - What's unclear: Measured cold-spawn prompt latency on the dev machine (login shell sourcing `.zprofile`/`.zshrc` can be slow with nvm/asdf).
   - Recommendation: Default ~4000 ms; measure real cold-spawn latency during E2E and tune. Make it a named constant (`READINESS_TIMEOUT_MS`) like `STOP_GRACE_MS`.

3. **Exact discard-vs-flush boundary after the marker.**
   - What we know: Discard probe bytes; flush real output.
   - What's unclear: Whether the marker round-trip and the shell's first prompt are cleanly separable in the byte stream.
   - Recommendation: E2E-capture the raw cold-spawn byte stream for zsh and bash, then tune the matcher and the discard window against real data before locking the implementation.

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|

## RESEARCH COMPLETE

**Phase:** 05.1 - TERM-05 startup-command auto-run
**Confidence:** HIGH (architecture + injection), MEDIUM (exact cross-shell probe sequence — E2E-tuned this phase)

### Key Findings
- **The invisibility problem is solved by architecture, not by terminal tricks.** PTY output flows `child.onData → PtyManager.send('pty:data') → renderer`. A probe is invisible (D-02) simply by buffering its bytes in main and never calling `send()` — the renderer literally never receives them. This is far more robust than hunting for a sequence xterm swallows.
- **The readiness crux = a one-shot output-gated round-trip probe.** Send a no-side-effect nonce marker (`: <nonce>\r`, POSIX no-op builtin — changes NO shell state, satisfying D-01) and wait to observe it round-trip in the *output* (not the echoed input — pexpect's documented trap). The echo proves zle/readline is live, which a fixed delay could only guess — this is precisely why the delay garbled cold spawns.
- **Injection is a main-side `child.write(cmd + "\r")`** — NOT the renderer's bracketed-paste `term.paste()` path (that exists to PREVENT auto-execute). CR (not LF) matches a real Enter and lands the command in zsh/bash history (SC1).
- **One hook in `create()` at the existing onData wiring (line ~281)** covers all three triggers (new/restart/Start-promotion, D-05); gate on `record.startupCommand?.trim()`. Refactor the existing `onData→send` into a shared `wireNormalOnData()`.
- **Timeout → flush buffer + bare shell + notice (D-04), never inject.** A `ReadinessProbe` seam mirroring `ShellDiscovery` carries per-shell knowledge; macOS now, Windows stub for Phase 8 (D-03).
