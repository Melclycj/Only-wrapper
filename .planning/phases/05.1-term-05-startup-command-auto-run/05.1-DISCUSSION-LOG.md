## Detection invasiveness

### How much footprint on the user's shell environment is acceptable?

| Option | Description | Selected |
|--------|-------------|----------|
| Passive observation only | Detect readiness purely by watching PTY output; never touch rc/env/hooks. Best fidelity, but the original cold-spawn failure lived here. | |
| One-shot transparent probe | Write a non-persistent readiness marker (send + await round-trip) to confirm the PTY line discipline is live, NO rc/env changes. Directly attacks the cold-spawn failure; momentary footprint. | ✓ |
| Full shell-integration hooks | Inject per-shell hooks (PROMPT_COMMAND/precmd, VS Code-style) for max robustness. Most reliable, but modifies the shell environment; furthest from "native, untouched shell." | |

**User's choice:** One-shot transparent probe
**Notes:** Aligns with Core Value (real fidelity, "sits on top of your shell") while giving research a real tool to fix the cold-spawn keystroke-loss problem rather than guessing by delay.

### How visible may the probe be in the terminal?

| Option | Description | Selected |
|--------|-------------|----------|
| Invisible — no artifact | Probe leaves no visible trace; user only ever sees their own command typed at a clean prompt. Constrains research to a no-echo technique. | ✓ |
| Minor transient marker OK | A brief marker line may appear before the command. Easier to implement; small cosmetic blemish in scrollback. | |

**User's choice:** Invisible — no artifact
**Notes:** Acceptance constraint on the implementation, not a mechanism choice. Highest fidelity.

## Ready-fail fallback

### If readiness can't be confirmed within a bounded timeout, what happens to the command?

| Option | Description | Selected |
|--------|-------------|----------|
| Give up + brief notice | Don't inject; leave a clean bare shell + a small non-intrusive notice. Saved command stays visible on idle card for manual run. Honors SC4. | ✓ |
| Give up silently | Leave a clean bare shell, no message. Simplest, but user may wonder why the command didn't fire. | |
| Best-effort inject anyway | Inject after max wait regardless. WARNING: the exact cold-spawn garble failure mode that descoped TERM-05 twice; risks SC4. | |

**User's choice:** Give up + brief notice
**Notes:** Timeout duration + notice surface left to planner discretion (DESIGN.md tokens).

## Trigger & opt-out

### Do you want a way to start a session WITHOUT auto-running its saved command?

| Option | Description | Selected |
|--------|-------------|----------|
| No — Start always runs it | If a command is saved, Start/Restart always runs it. Smallest scope; matches the agent-workflow core use case. Escape hatch can be a Phase 6 deferred idea. | ✓ |
| Yes — add "start without command" | Secondary action (context-menu / modifier on ▶) spawns a bare shell in the saved cwd/shell, skipping the command. Covers "just want a shell here" without editing the profile. More UI surface. | |

**User's choice:** No — Start always runs it
**Notes:** Trigger rule itself (auto-run on every spawn with a non-empty command; dormant restore-on-launch never auto-runs per SC5) was confirmed as fixed by the SCs + the single spawn path, not re-debated.
