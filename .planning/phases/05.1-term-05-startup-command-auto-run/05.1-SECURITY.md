### T-05.1-01 — startupCommand reaching/written to the PTY (and the timeout path) — ACCEPT — CLOSED

Disposition is `accept` (the command is the user's OWN saved string run in their OWN
shell — same trust boundary as typing it natively). Verification confirmed all four
acceptance conditions hold in code:

1. **Injection only via `record.startupCommand`.** The sole command-write site is
   `child.write(cmd + '\r')` at `src/main/pty-manager.ts:383`, where
   `cmd = record.startupCommand?.trim()` (`pty-manager.ts:338`). The other two
   `*.write` sites are the no-op probe marker (`pty-manager.ts:389`, no user data) and
   the user-keystroke forwarder (`pty-manager.ts:653`, the existing `ptyWrite` IPC).
   `record.startupCommand` is set ONLY at `pty-manager.ts:289` (carry-through from the
   prior record) and `pty-manager.ts:766-767` (the `updateProfile` handler, gated on
   `typeof fields.startupCommand === 'string'`).
2. **Gated on `.trim()`.** `if (!cmd)` (`pty-manager.ts:339`) skips the probe entirely
   for empty/whitespace commands; only the non-empty branch can ever reach the write.
   Unit-asserted: `child.write` is NOT called with `'   \r'` or `'\r'`
   (`src/main/__tests__/readiness-probe.test.ts:302-303`).
3. **No renderer auto-run write path.** `IdleCard.tsx:8-10` documents and enforces that
   the saved command is displayed read-only with NO `window.api.ptyWrite`. The only
   renderer `ptyWrite` calls are live user keystrokes (`SessionView.tsx:183`,
   `TerminalPane.tsx:141`). In the renderer, `startupCommand` flows only into the edit
   modal → `onSaveProfile` → `ptyUpdateProfile` persist path, never into a write.
4. **Timeout branch never injects (safe-by-default).** The `setTimeout` branch
   (`pty-manager.ts:397-418`) flushes the buffered shell output and emits a notice but
   contains NO `child.write` of the command. Unit-asserted: on timeout the buffer is
   flushed and `child.write` is NOT called with the command
   (`readiness-probe.test.ts` timeout assertion, GREEN per 05.1-03 SUMMARY).

Same trust boundary, no new privilege, no renderer write path, safe-by-default on
failure — the accept disposition is correctly realized in code.

### T-05.1-02 — the probe marker string — MITIGATE — CLOSED

`buildPosixProbe(nonce)` (`src/main/readiness-probe.ts:65-77`) builds
`marker: \`: ${nonce}\r\`` — a FIXED `: ` POSIX no-op builtin plus a self-generated
nonce. The nonce is generated inside `MacReadinessProbe.forShell()`
(`readiness-probe.ts:89`) via `crypto.randomBytes(8).toString('hex')` wrapped in the
`__JW_READY_<hex>__` sentinel. NO user-controlled data (no `startupCommand`, no `cwd`,
no `shell`) is interpolated into the marker. `forShell(shellPath)` discards its
argument (`void shellPath`, `readiness-probe.ts:88`). The module imports only
`node:crypto` — no electron, no node-pty, no user input source.

### T-05.1-03 — nonce / command / buffered bytes in logs — MITIGATE — CLOSED

The probe/inject block performs lifecycle-only logging. The two `console.log` calls in
the spawn/probe region log only static lifecycle text:
`[pty] spawned ${shell} pid=... (session ${id})` (`pty-manager.ts:311`) and
`[pty] readiness probe timed out (session ${id}) — auto-run skipped`
(`pty-manager.ts:417`). Neither interpolates `cmd`, `record.startupCommand`, `buffer`,
the nonce, or `probe.marker`. `readiness-probe.ts` contains no `console.*` at all. The
buffered (nonce-bearing) bytes are discarded on match (`pty-manager.ts:374-376`) and
flushed-as-PTY-data-but-never-logged on timeout (`pty-manager.ts:404`). V7 rule upheld.

### T-05.1-04 — the ready-fail notice payload — MITIGATE — CLOSED

The notice is a FIXED module-level literal `READINESS_FAIL_NOTICE`
(`pty-manager.ts:87-88` = `"Startup command didn't auto-run — shell wasn't ready in
time."`). The timeout branch sends exactly `{ id, status: liveStatus, notice:
READINESS_FAIL_NOTICE }` (`pty-manager.ts:411-415`) — no `startupCommand`, nonce, or
buffer is interpolated into the notice. The renderer renders it as a fixed dim line
(`SessionView.tsx:225-227`).

> **Deferred hardening (WR-04, advisory — NOT a blocker):** the renderer writes
> `p.notice` to the terminal with ANSI wrappers and trusts the channel payload. Today
> main only ever sends the fixed literal, so no untrusted/control-char content can
> reach `term.write`. Renderer-side control-character sanitization of `notice` is a
> documented defense-in-depth deferral (REVIEW.md WR-04) and is out of scope for this
> phase's declared mitigation, which is satisfied: the notice main sends is a fixed
> literal with zero interpolation.

### T-05.1-05 — bridge surface drift — MITIGATE — CLOSED

The notice rides the EXISTING `onPtyStatus` / `pty:status` channel via an additive
optional `notice?: string` field on `PtyStatusPayload` (`src/shared/api-types.ts:66-80`).
No new contextBridge key was added: `EXPECTED_API_KEYS` remains the same 18-key set
(`src/main/window-config.ts:76-95`), and the preload exposes exactly those 18 methods
with no new key and no raw `ipcRenderer` (`src/preload/index.ts:36-178`). The
lockstep guard still asserts `Object.keys(exposed).sort() === EXPECTED_API_KEYS.sort()`
and `not.toHaveProperty('ipcRenderer')` (`src/shared/__tests__/security.guard.test.ts:48-52`),
GREEN with no change. Pitfall 5 (bridge drift) avoided by design.
