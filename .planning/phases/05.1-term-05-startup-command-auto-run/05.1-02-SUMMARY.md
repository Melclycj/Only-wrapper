# Phase 05.1 Plan 02: create() Probe-then-Inject Happy Path Summary

The core vertical slice of TERM-05: after a PTY spawns with a non-empty `startupCommand`, `create()` runs the invisible one-shot readiness probe and — once the shell genuinely accepts input — injects the command as `cmd + '\r'` so it runs and lands in shell history (SC1). Probe bytes are buffered in main and never forwarded to the renderer (D-02). Empty/whitespace commands skip the probe entirely and start a normal bare shell (SC2/TERM-03). The single hook covers all three spawn entry points (D-05).

## What Was Built

**Task 1 — `src/main/pty-manager.ts` probe-then-inject hook + `wireNormalOnData`:**
- Added `import { selectReadinessProbe } from './readiness-probe'`.
- Extracted the inline `child.onData(...)` forwarder into a private `wireNormalOnData(id: LogicalId, child: IPty): void` (the shared UTF-8 pass-through forwarder; SC4 no binary re-encoding comment carried over). Called on BOTH the empty-command path AND the post-probe resume path.
- Replaced the old inline forwarder block (placed BEFORE the unchanged `child.onExit(...)`) with the gated probe hook:
  - `const cmd = record.startupCommand?.trim()`.
  - **Falsy `cmd`** → `this.wireNormalOnData(id, child)` and skip the probe (SC2/TERM-03).
  - **Non-empty `cmd`** → `const probe = selectReadinessProbe(process.platform).forShell(record.shell)`; a `let buffer = ''` / `let settled = false`; a transient interceptor `const offProbe = child.onData(...)` that BUFFERS bytes (never `this.send`) until `probe.matches(buffer)`, then `settled = true`, `offProbe.dispose()`, DISCARDS the buffer, `this.wireNormalOnData(id, child)`, and `child.write(cmd + '\r')` once (SC1). Then `child.write(probe.marker)` sends the no-side-effect nonce probe.
  - A `settled` guard forwards any byte that races in between the match and the listener swap so no post-ready chunk is lost.
- **Plan 03 seams:** two clearly-labeled `// TODO(Plan 03 / D-04)` comments mark (a) where to clear the timer on match and (b) where the `setTimeout` flush-and-notice branch goes (FLUSH `buffer` via `this.send`, `wireNormalOnData`, emit ready-fail notice, NEVER best-effort inject). No timer implemented here.
- **Security V7 (T-05.1-03):** lifecycle logging only — the command, the probe nonce, and the buffered (nonce-bearing) bytes are NEVER logged (grep-confirmed clean).

## Confirmations (no code — per the plan)

- **Restart-separator ordering (A6/Pitfall 2):** the `— restarted HH:MM —` separator is written RENDERER-side in `SessionView.tsx:219` on the `onPtyStatus('running')` event, so it never passes through the main-side probe buffer and cannot be swallowed by the probe. Resolves the RESEARCH ordering concern favorably.
- **SC5/D-06 (dormant restore never auto-runs):** `restart()` and the dormant Start-promotion both funnel through `create()`; a dormant-restored record lives in `dormantRecords` with NO pty until `create({id})` promotes it, so a restore-on-launch never reaches this hook — there is nothing to inject into. Structurally satisfied with no code change.
