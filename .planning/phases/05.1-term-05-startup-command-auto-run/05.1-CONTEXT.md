# Phase 05.1: TERM-05 startup-command auto-run - Context

**Gathered:** 2026-06-06
**Status:** Ready for planning

<domain>
## Phase Boundary

Auto-run a session's optional saved **startup command** into the PTY once the
shell is **genuinely ready** — written as if the user typed it followed by Enter,
**visible, landing in shell history** — so starting (or restarting) an agent
session such as `claude --rc` relaunches the tool in its working directory
without the user retyping it.

This **un-defers TERM-05**, which was descoped twice:
1. From **Phase 3** — the naive settle-delay injection was unreliable on cold
   first spawn (lost/garbled keystrokes).
2. From **Phase 5** — re-homed to its own increment because it needs
   shell-ready-detection that the persistence phase didn't cover.

**Requirements covered:** TERM-05.

**Core new behavior:** opens the boundary that Phase 5 deliberately held closed —
`IdleCard` currently *displays* `startupCommand` read-only and never executes it.
This phase makes the command actually run, hooked into the single spawn path,
gated on a reliable readiness signal.

**Explicitly NOT in this phase:**
- **Windows ready-detection** (PowerShell/CMD/Git Bash/WSL) — built behind a seam
  now, real implementation + verification at **Phase 8** (Windows isn't runnable
  on the dev machine; mirrors Phase 5's `ShellDiscovery` D-07).
- **"Start without running the command"** escape hatch — deferred to **Phase 6**.
- **App-level agent conversation/state resume (Layer C)** — out of scope project-wide;
  power users get it via their own `claude --continue` startup command.

</domain>

<decisions>
## Implementation Decisions

### Shell-Ready Detection (the twice-descope crux)
- **D-01: One-shot transparent probe — non-persistent, NO rc/env changes.** The app
  MAY write a non-persistent readiness marker (send + await round-trip) to confirm the
  PTY's line discipline is live before injecting. It MUST NOT inject persistent shell
  hooks (`PROMPT_COMMAND`/`precmd` sentinels), modify rc files, or alter the user's
  shell environment. This rules out BOTH the naive blind fixed-delay (the original
  cold-spawn failure) AND full shell-integration. The exact probe technique is the
  researcher's job, constrained by D-02. Rationale: Core Value = real terminal fidelity
  ("behaves exactly like a native terminal; not a new shell — sits on top of the user's
  existing shells") — momentary, gone-before-the-command footprint is acceptable;
  persistent environment changes are not.
- **D-02: The probe MUST be INVISIBLE — no visible artifact.** Acceptance constraint
  (not a mechanism choice): the only thing the user ever sees is their own startup
  command typed at a clean prompt. Research must pick a no-echo technique (a swallowed
  control/OSC sequence, or a marker cleared before the command runs). No junk line in
  scrollback.

### Platform Scope
- **D-03: macOS shells (zsh/bash) now, behind a detection seam; Windows at Phase 8.**
  Fully implement + verify ready-detection for the POSIX shells on macOS (where dev/test
  happens). Put detection behind a seam **mirroring Phase 5's `ShellDiscovery` (D-07)**.
  Windows ready-detection (PowerShell/CMD/Git Bash/WSL) lands when Windows is runnable at
  Phase 8 — a clean drop-in. **SC1–SC5 are proven on macOS this phase.** Matches the
  project's "test fidelity where it's built" principle and the existing
  `shell-resolver.ts` macOS-now/Windows-Phase-8 pattern.

### Failure Handling
- **D-04: On ready-timeout → give up + brief notice (never inject-anyway).** If readiness
  can't be confirmed within a bounded timeout, do **NOT** inject — leave a clean, usable
  bare shell (a bare shell beats garbled keystrokes; protects **SC4**). Surface a small
  non-intrusive notice (e.g. "Startup command didn't auto-run — shell wasn't ready in
  time"); the saved command stays visible on the idle card for manual run. Best-effort
  inject-after-timeout was explicitly rejected — it IS the cold-spawn garble failure mode
  this phase exists to fix. Timeout duration + notice surface = Claude's discretion.

### Trigger Surface
- **D-05: Auto-run fires on EVERY PTY spawn whose record has a non-empty `startupCommand`.**
  That is: **Start ▶** (dormant promotion), **Restart**, and any new session created
  already carrying a command. One hook on the single spawn path in
  `PtyManager.create()` covers all three.
- **D-06: Dormant restore-on-launch NEVER auto-runs (SC5) — satisfied structurally.**
  Restored sessions spawn no PTY (Phase 5 dormant model: restore = profiles-only); with
  no PTY there is nothing to inject into. The command runs only when the user explicitly
  Starts the session. No special-case code — it falls out of the existing model.
- **D-07: No opt-out this phase — Start always runs a saved command.** If a command is
  saved, Start/Restart always runs it (that's the point of saving it). The "start without
  running the command" (bare shell in the same cwd/shell) escape hatch is **deferred to
  Phase 6**.

### Locked from prior phases (carried forward — NOT re-decided here)
- **Injection model:** the command is written into the PTY **as if the user typed it +
  Enter** — VISIBLE, lands in shell history (Phase 3 D-05 historical model + SC1).
- **Re-run on restart** (SC3 / Phase 3 D-03), after the `— restarted HH:MM —` separator.
- **Empty / whitespace `startupCommand` → no injection;** normal bare-shell mode stays
  intact (SC2 / TERM-03).

### Claude's Discretion (guided by SCs + the D-01/D-02 constraints + research)
- The exact invisible round-trip probe technique per POSIX shell (within D-01/D-02).
- The readiness/probe **timeout duration** — a sensible bound (e.g. low single-digit seconds).
- The ready-fail notice's **visual treatment + surface** (DESIGN.md tokens — transient
  toast in the session, idle-card hint, or header notice).
- Where the auto-run + detection logic sits relative to `create()`'s post-spawn `onData`
  wiring; the detection **seam interface shape** (mirror `ShellDiscovery`).
- Any bridge surface additions — extend the typed `contextBridge` + `EXPECTED_API_KEYS` +
  `security.guard.test.ts` in lockstep; never raw `ipcRenderer`. Likely minimal (mostly
  main-side); a renderer signal may be needed only for the D-04 notice.

</decisions>

- **`PtyManager.create()`** — the single spawn path; all spawn entry points converge here,
  so ONE post-spawn hook covers Start / Restart / create-with-command (D-05).
- **`SessionRecord.startupCommand`** — already persisted (Phase 5) and edited via the form
  (Phase 4); read it at spawn time. No type reshaping needed.
- **`IdleCard`** — already shows the saved command read-only; the D-04 manual-run fallback
  leans on it staying visible.
- **`shell-resolver.ts` seam** — the established macOS-now / Windows-Phase-8 pattern to
  mirror for the detection seam (D-03).
- **Phase 3 restart path** (`— restarted HH:MM —` separator + re-spawn) — auto-run re-runs
  naturally on this path (SC3); the command lands after the separator.

### Established Patterns
- contextBridge-only renderer↔main seam; node-pty + all PTY I/O in **main only** (injection
  logic belongs in main, not the renderer); per-task atomic commits;
  `security.guard.test.ts` asserts the exact bridge key surface — any new IPC updates
  `EXPECTED_API_KEYS` + the guard in lockstep.
- Platform-specific behavior behind a seam: macOS implemented + tested now, Windows contract
  stubbed for Phase 8 (`shell-resolver.ts` / `ShellDiscovery` precedent).
- Main is the authoritative record/spawn owner.

### Integration Points
- **Spawn:** `create()` post-spawn → run the invisible ready-probe → on ready, write
  `startupCommand` + Enter into the PTY (only if non-empty).
- **Timeout:** bounded wait → on miss, skip injection + emit the D-04 notice to the renderer.
- **Restart:** existing restart path reuses the same auto-run hook (SC3).
- **Dormant restore:** no PTY spawned on launch → no injection (SC5) — nothing to build,
  just don't auto-spawn (already true).
