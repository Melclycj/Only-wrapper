<objective>
Deliver the create/edit + identity vertical slice end-to-end: right-click a sidebar row -> context menu -> Edit -> a modal form whose name/icon apply live and whose cwd/shell/startupCommand persist to main (and apply on the next restart), an icon picker (emoji grid + free-text + color badge-with-initial), and a slim identity header above the active terminal. After this plan a user can produce the canonical `🛋️ Parlour Claude RC` session profile through real UI.

Per D-01, the existing "+ Add session" quick-add (instant-spawn a live `Session N` with the default 🖥️ icon — `session-add.ts`) stays exactly as-is; this plan adds NO create-first path. The modal is an EDIT form that customizes an already-spawned session. Per D-04 it is a MODAL dialog reusing the `ConfirmModal` overlay pattern.

Purpose: Satisfy SESS-01/02/03/04 + IDENT-03 — the phase's biggest user-visible payoff. The form is the only place all five SessionRecord identity fields become functional; live name/icon edits preserve logicalId (SESS-04 / IDENT-02); restart-applied fields use the Plan 01 `ptyUpdateProfile` seam (SESS-01 "all fields functional").
Output: SessionEditModal, IconPicker, ContextMenu, IdentityHeader (new components), the Sidebar color-badge-with-initial change + row-level onContextMenu, and the SessionManager wiring that hosts the modal/menu/header state and mirrors live edits to main.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| renderer form input -> window.api.ptyUpdateProfile -> main | User-typed name/icon/cwd/shell/startupCommand cross to the main record store |
| DOM events (right-click, paste) -> renderer components | Untrusted UI events drive menu/modal/picker state |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | edited fields reaching main | mitigate | main-side `updateProfile` (Plan 01) id-validates + type-guards every field before writing; renderer is not trusted |
| T-04-05 | Spoofing/Integrity | edit creating a new session id | mitigate | onSaveLive only `setSessions`-maps the existing row + mirrors via ptyUpdateProfile; never ptyCreate/newLogicalId (SESS-04/IDENT-02) — grep-verified |
| T-04-06 | Information Disclosure | logging form values (paths/commands) | accept | No new logging added; existing "never log raw PTY data / form secrets" rule (V7) — components do not console.log field values |
| T-04-04 | Misuse | startupCommand captured then executed | mitigate | Form only stores startupCommand via ptyUpdateProfile; no PTY-write path exists (TERM-05 deferred, enforced in Plan 01) |
| T-04-SC | Tampering | npm installs | mitigate | N/A — Phase 4 installs zero packages |
</threat_model>
