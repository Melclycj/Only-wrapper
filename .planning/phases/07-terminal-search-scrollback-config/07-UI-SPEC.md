# Phase 7 — UI Design Contract

> Visual and interaction contract for Phase 7: Terminal Search + Scrollback Config. Generated by gsd-ui-researcher; to be verified by gsd-ui-checker.
>
> **This phase extends existing surfaces — it introduces NO new design system and NO new runtime UI dependency.** All tokens are pulled verbatim from `.planning/DESIGN.md` (the warm "parlour" system) and the already-shipped `:root` token set in `src/renderer/terminal.css`. The one new package this phase adds, `@xterm/addon-search`, is a pure-JS xterm addon (find logic + DOM/canvas decorations) — **not** a UI registry, component library, or styling dependency. The shadcn gate is **not applicable**: the project is mid-build (phase 7 of 9) on a locked, hand-rolled CSS custom-property design system; introducing shadcn/Tailwind here would contradict DESIGN.md's locked visual authority and the six prior shipped phases.
>
> **New surfaces this phase owns:**
> 1. **In-terminal search bar overlay** (SC1 / SC3 / TERM-10, D-01..D-03) — a VS Code-style top-right overlay over the active session's terminal: text input + prev/next nav + live "N of M" match count + "Aa" case-sensitivity toggle + close. Opened by the find chord (Cmd+F mac / Ctrl+F win) over the existing `'session:switch'` channel; Esc dismisses; when closed it never touches terminal input.
> 2. **Preferences modal** (SC2 / TERM-11, D-08) — a lightweight, extensible Preferences shell launched from a sidebar gear icon, reusing the shipped modal idiom (`ConfirmModal` / `SessionEditModal` / `IconPicker`). Holds only the scrollback setting now (numeric stepper, default 5000, range 1000–50000 per D-04) but is structured to grow.
> 3. **Sidebar gear control** (TERM-11, D-08) — the ⚙ entry point that opens Preferences, placed so it is reachable in both expanded and collapsed-rail modes.
>
> Pre-locked decisions D-01..D-08 are LOCKED contract inputs — formalized here, not relitigated. Items the CONTEXT `### Claude's Discretion` block delegated to this agent are resolved with DESIGN.md-grounded defaults and tagged **(default — Claude's Discretion per CONTEXT)**.

## Color

The full palette is `:root` in `terminal.css` (DESIGN.md subset). This phase introduces **no new color** — every value below is pulled verbatim from the shipped token set / the shipped `oklch()` accents already in `terminal.css`. The 60/30/10 split maps the parlour system; the search bar and Preferences modal are calm warm chrome, accent reserved narrowly.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--surface` `#ffffff` + `--bg-sunk` `oklch(0.955 0.01 85)` | Search-bar overlay card body (`--surface`); Preferences `.modal-dialog` body (`--surface`); active/hover wash on controls (`--bg-sunk`). |
| Secondary (30%) | `--term-bg` `#1e232c` (gutter the overlay floats on) + `--ink`/`--ink-soft`/`--ink-faint` ink ramp + `--line`/`--line-soft` borders | Terminal gutter behind the overlay; all text, the count label, the helper copy, hairlines; neutral resting state of search nav/Aa/close + gear (`--ink-soft`). |
| Accent (10%) | blue `oklch(0.62 0.14 248)` (the single shipped focus/"go" accent) | Reserved list below. |
| Match highlight | search decorations (see below) | Reserved exclusively for terminal search-match highlighting (not chrome). |

**Search-match decorations (D-01 highlight-all + active match — required for the decorations-gated `onDidChangeResults` count per 07-RESEARCH Pitfall 1).** These are `ISearchOptions.decorations` colors applied to terminal cells, NOT chrome. Chosen to read on the dark `--term-bg` terminal background and harmonize with the amber/blue accent family **(default — Claude's Discretion per CONTEXT)**:

| Decoration field | Value | Role |
|------------------|-------|------|
| `matchBackground` (all matches) | `oklch(0.66 0.15 60 / 0.32)` | amber wash on every match — the parlour attention hue at low alpha so highlighted matches read as "found here" without shouting |
| `activeMatchBackground` (current match) | `oklch(0.66 0.15 60 / 0.7)` | the same amber at high alpha so the current match stands out from the rest |
| `matchOverviewRuler` (REQUIRED string) | `oklch(0.66 0.15 60)` | overview-ruler tick for all matches (xterm requires this non-optional field — 07-RESEARCH Pitfall 1) |
| `activeMatchColorOverviewRuler` (REQUIRED string) | `oklch(0.62 0.14 248)` | blue overview tick for the active match (the shipped "current/focus" accent), distinct from the amber all-match ticks |

> Rationale: amber `oklch(0.66 0.15 60)` is the DESIGN.md "attention / waiting-for-you" hue. In the chrome it is reserved exclusively for the agent-state "Waiting for you" badge (Phase 6 lock). Here it is used **only inside the terminal viewport as a search-match wash** — a different surface (terminal cells, not a status chip) and a different semantic (a literal "this text matched"), so it does not collide with the chrome reservation. The active-match overview tick uses the blue "current/focus" accent to mirror how blue marks the focused/active affordance everywhere else.

**Accent reserved for (explicit — never "all interactive elements"):**
- **Input / button focus ring** `oklch(0.62 0.14 248)` (blue, 2px, `outline-offset: 1px`) — the single, consistent focus affordance already shipped. The search input, search nav/Aa/close buttons, the gear, the scrollback numeric input, and the Preferences Save button all inherit it verbatim.
- **Aa case-toggle ACTIVE state** — when case-sensitivity is ON, the toggle reads in the blue accent (`border-color` + `color` `oklch(0.62 0.14 248)`, `background oklch(0.62 0.14 248 / 0.08)`) mirroring the shipped `.row-control-start` "active/engaged" treatment. When OFF it is neutral `--ink-soft`. This is the ONE new accent usage on chrome this phase — a pressed-state telegraph, not decoration.
- **Search-match decorations** — amber (all) + blue (active overview tick) as tabled above, confined to the terminal viewport.
- **Preferences Save button** — the primary "confirm" affordance; reuses the shipped `.modal-btn-confirm` shape BUT in the **blue** "go" accent (not the destructive red of the Close-confirm), since saving a preference is constructive, not destructive (see Copywriting / Interaction Contract 2). `oklch(0.62 0.14 248)` fill, white text.

**No destructive color this phase.** Saving a scrollback value, opening/closing the search bar, and toggling case-sensitivity are all non-destructive — no red ramp, no confirm dialog. (Lowering scrollback trims off-screen rows per D-06, but that is expected, reversible-by-raising, and not framed as destructive — see Copywriting.)

## Interaction Contracts

> The surfaces this phase introduces, spec'd prescriptively for the executor. Mirrors the 06-UI-SPEC precedent's "Interaction Contracts" shape.

### 1. In-terminal search bar overlay (SC1 / SC3 / TERM-10 — D-01, D-02, D-03)

A VS Code-style find overlay over the active session's terminal. It is a sibling DOM overlay (NOT inside `.xterm`), absolute-positioned over `.viewport-stack`, mounted per `SessionView`, visible only when that session's `searchOpen` state is true.

- **Placement (default — Claude's Discretion per CONTEXT):** `position: absolute; top: 8px; right: 8px; z-index: 5` over the active `.session-view` / `.viewport-stack`. Top-right is the VS Code convention and keeps the overlay clear of the prompt line (bottom) and the identity header (above the stack). It floats as a small `--surface` card with `--line` border, `radius: 10px`, the rail-tooltip shadow (`0 6px 18px oklch(0.32 0.012 70 / 0.14)`), inner pad `6px 8px`, control gap `4px`. It renders ONLY for the active session's view (hidden panes never show it).
- **Layout (left→right):** `[ mono input ]  [ "N of M" count ]  [ ‹ prev ]  [ › next ]  [ Aa ]  [ ✕ ]`. Input flexes to `min-width: 200px`; the count + 4 buttons are fixed `.row-control` 24×24 footprints (except the count, a 12px label). Single row, no wrap.
- **Open (D-02 / D-03 — find chord, zero new bridge key):** Cmd+F (macOS) / Ctrl+F (Windows) is matched MAIN-side in `before-input-event` → `matchSearchKey` returning a `{ kind: 'search' }` `SwitchIntent`, dispatched over the EXISTING `'session:switch'` channel (mirrors the Clear-chord precedent; `EXPECTED_API_KEYS` stays **19**). The renderer's `onSwitchSession` branches `'search'` exactly like `'clear'`, toggling the active session's `searchOpen`. **The chord never reaches xterm/PTY** (preventDefault main-side), so macOS `Cmd+F` does not steal a PTY key and Windows `Ctrl+F` is intercepted before the shell. On open, the input auto-focuses and selects any prior query.
- **Typing / navigation:** the input's `onKeyDown` handles **Enter → next match**, **Shift+Enter → previous match**, **Esc → close**, and calls `stopPropagation()` so keystrokes go to the DOM input, NOT the terminal (07-RESEARCH Pitfall 3). Each query change / nav calls `searchAddon.findNext|findPrevious(query, opts)` where `opts.decorations` is ALWAYS present (required for highlight-all AND for the count event to fire — Pitfall 1). Search wraps at buffer ends (SearchAddon default — adopted per CONTEXT discretion).
- **"N of M" count (D-01):** subscribe to `onDidChangeResults(({ resultIndex, resultCount }) => …)`. Render per the Copywriting table: `count === 0` → "No matches"; `resultIndex === -1` (over threshold) → "{count}+ matches"; else → "{resultIndex + 1} of {count}". The label sits between the input and the nav buttons so it reads as "where am I in the results".
- **Case-sensitivity toggle "Aa" (D-01):** a `.row-control`-footprint button with `aria-pressed`. OFF = neutral `--ink-soft`. ON = blue engaged treatment (border + text `oklch(0.62 0.14 248)`, bg `oklch(0.62 0.14 248 / 0.08)`). Toggling re-runs the current query with `caseSensitive` flipped so the count + highlights update immediately.
- **Close / dismiss (SC3):** the ✕ button OR Esc closes the bar → sets `searchOpen=false`, calls `searchAddon.clearDecorations()`, and returns focus to the terminal. **When closed the overlay is unmounted (or `hidden`), so it cannot intercept any keystroke — terminal input behaves exactly as before (SC3 locked).** The find chord is the only thing that re-opens it.
- **States to lock (all 7):**
  1. **closed** — overlay absent; terminal input unaffected (SC3).
  2. **open-empty** — visible, input focused, placeholder "Search this terminal", count area empty, no decorations.
  3. **typing-with-matches** — decorations highlight all matches (amber wash), active match strong amber, count "{i} of {M}".
  4. **typing-no-match** — count "No matches" (`--ink-faint`, not red), no decorations, input border stays neutral (no error styling — an empty result is not an error).
  5. **over-threshold** — count "{M}+ matches" (the `resultIndex === -1` sentinel), decorations still highlight the capped set, no current-index shown.
  6. **case-toggle-active** — Aa in the blue engaged state; results re-computed case-sensitively.
  7. **input focus ring** — the input shows the 2px blue `:focus-visible` outline (shipped focus affordance), `outline-offset: 1px`.
- **Lifecycle (07-RESEARCH Pitfall 4):** `SearchAddon` loads ONCE per `Terminal` in the mount effect (alongside Fit/WebLinks/Unicode11/WebGL) and is disposed in the existing cleanup before `term.dispose()`; the `onDidChangeResults` subscription is unsubscribed in the same cleanup. No new visible chrome from lifecycle.
- **`data-testid` hooks (for the smoke/manual driver):** `search-bar` (overlay), `search-input`, `search-count`, `search-prev`, `search-next`, `search-case` (Aa), `search-close`.

### 2. Preferences modal (SC2 / TERM-11 — D-08)

A lightweight, EXTENSIBLE Preferences shell launched from the sidebar gear, reusing the shipped `ConfirmModal` / `SessionEditModal` overlay+scrim+Esc+focus-on-open a11y skeleton (clone the skeleton; do NOT generalize `ConfirmModal`). It holds only the scrollback field now but is laid out as a list of settings rows so future settings drop in without re-architecture.

- **Shell:** `.modal-overlay` scrim (click = dismiss) + `.modal-dialog` card (`width: min(420px, calc(100vw - 48px))`, `role="dialog"`, `aria-modal="true"`, `aria-labelledby` the title). Title `.modal-title` "Preferences". Body is a vertical stack of setting fields (one now). Esc dismisses; focus lands on the scrollback input on open (mirrors `SessionEditModal`).
- **Extensibility framing (D-08):** the body is a `<div className="prefs-body">` containing one `.edit-field`-style group per setting. Adding a setting = appending another field group — no layout rewrite. Document this so the executor builds the shell, not a one-off.
- **Scrollback field (D-04):** an `.edit-field` with:
  - **Label:** `.edit-label` "Scrollback lines".
  - **Control:** a numeric input — `<input type="number" min={1000} max={50000} step={1000}>` styled with `.edit-input` verbatim (`8px 10px`, radius 10px, `--surface`, `--line`, Nunito 14px, blue `:focus-visible` ring). The native number-input stepper arrows provide increment/decrement; `step={1000}` gives sensible nudges. `data-testid="pref-scrollback"`.
  - **Helper:** `.idle-card-helper`-style line (`--ink-faint`, 12px/400) — "How many lines of history each terminal keeps. Between 1,000 and 50,000." + the live-apply reassurance sub-hint.
  - **Validation / clamp (D-04):** the value is clamped to [1000, 50000] via the pure `clampScrollback` helper (07-RESEARCH) before it is applied or persisted. A typed out-of-range value snaps to the nearest bound on commit (blur / Save), and the clamp hint "Keep this between 1,000 and 50,000 lines." reads in `--ink-faint` (guidance, not red error). The input is never left in an invalid state — there is no error border, no blocked-save modal.
  - **Default:** 5000 (the field shows the persisted value on open, falling back to 5000 — D-04).
- **Apply + persist (D-05 / D-07):** **(default — Claude's Discretion per CONTEXT)** prefer **live-apply-on-commit**: on input commit (debounced change or blur), `SessionManager.handleSetScrollback(N)` (a) fans the clamped value to every mounted `term.options.scrollback = N` (D-05, no IPC), (b) seeds new `SessionView`s via the prop, and (c) persists through the existing `persistUiState` path (no new bridge key — rides `ui`, `EXPECTED_API_KEYS` stays 19). The modal then has a single **"Done"** dismiss (`.modal-btn`, neutral) rather than a Save/Cancel pair. If the planner instead keeps an explicit draft-then-Save model, the Save button uses the blue "go" treatment (NOT the red `.modal-btn-confirm`) and Cancel uses the shipped `.modal-btn-cancel` — but live-apply is the recommended, lower-friction default for a single self-applying setting.
- **`data-testid` hooks:** `preferences-modal` (overlay), `pref-scrollback` (input), `preferences-done` (dismiss) [or `preferences-save` / `preferences-cancel` if draft-model].

### 3. Sidebar gear control (TERM-11 — D-08)

The ⚙ entry point that opens Preferences.

- **Placement (default — Claude's Discretion per CONTEXT):** in the sidebar's pinned-control row, as a sibling of the existing `.sidebar-collapse` chevron at the TOP of the `<nav>`. Placing it beside the chevron makes it reachable in BOTH expanded and collapsed-rail modes (the chevron is already styled to stay visible and re-center when collapsed — `terminal.css:317-319`), so the gear inherits that proven dual-mode reachability without bespoke collapsed-rail handling. It does NOT go at the bottom near "+ Add session" (that footer is the create affordance; mixing global-settings there muddies the two).
- **Footprint:** reuse the shipped `.sidebar-collapse` shape verbatim — **28×28, radius 8px**, transparent border, `--ink-soft` resting → `--bg-sunk` bg + `--ink` on hover, blue 2px `:focus-visible` ring. The two pinned controls (chevron + gear) sit in a small top row (`display: inline-flex; gap: 4px`); when collapsed they stack/center like the chevron already does.
- **Glyph + copy:** **⚙** (U+2699), `aria-label="Preferences"`, `title="Preferences"`. `data-testid="open-preferences"`.
- **Hover / focus:** identical to `.sidebar-collapse` — warms to `--bg-sunk` / `--ink` on hover; blue focus ring on `:focus-visible`. No accent color at rest (it is a neutral utility control, not a "go" affordance).
- **Behavior:** click → opens the Preferences modal (sets a SessionManager-level `preferencesOpen` state). Keyboard: native `<button>`, Enter/Space activates.

### 4. Scrollback live fan-out (SC2 / D-05) — behavior, no new chrome

Listed for completeness; this is a behavioral contract with no new visible surface beyond the Preferences field above.

- The active scrollback value is owned by `SessionManager` (boot-read, defaulting to 5000 until the persisted value is read per 07-RESEARCH Open Q1), passed as a prop to every `SessionView`.
