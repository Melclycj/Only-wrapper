<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| renderer → main (IPC) | the scrollback value crosses the bridge via `persistUiState`; the renderer payload is untrusted. |
| main → disk (lowdb) | only main writes the validated/clamped scrollback to the store. |
| main → renderer (getUiState) | the boot-read returns only the validated UI prefs object (no fs handle, no raw state). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Tampering | `persistUiState({ scrollback })` → `setUiState` | mitigate | The scrollback is validated finite + `clampScrollback`-bounded IN MAIN (Plan 01, T-05-01) before any disk write — a forged/out-of-range renderer payload clamps or no-ops. The renderer `clampScrollback` (this plan) is a UX convenience only; the main clamp is the security boundary (defense in depth). |
| T-07-06 | Information disclosure | `getUiState` read channel | mitigate | `getUiState` returns ONLY the in-memory validated UI prefs (`{ collapsed?, bounds?, scrollback? }`) via a shallow copy — no fs handle, no raw store object, no session secrets (mirrors `pickDirectory` returning only a string path). The renderer-never-touches-disk invariant holds — the boot-read is a narrow read-only accessor. |
| T-07-02 | Elevation | bridge surface | mitigate | This plan adds ZERO new bridge keys — scrollback persistence rides the existing `persistUiState` (widened payload, frozen by Plan 01) and the boot-read uses the `getUiState` key Plan 01 already added. The surface stays at 20; `security.guard.test.ts` is untouched and GREEN. |
| T-07-07 | Denial of service | scrollback fan-out at many sessions | accept | `term.options.scrollback = N` is a cheap scalar assignment guarded to fire only on actual change (Pitfall 5); negligible at the project's ~50-session ceiling. No re-fit triggered. [ASSUMED — not load-tested; documented in RESEARCH A1.] |
</threat_model>
