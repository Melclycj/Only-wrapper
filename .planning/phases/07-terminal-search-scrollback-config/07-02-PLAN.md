<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| search input (DOM) → PTY | the search `<input>` is a DOM overlay; its keystrokes must NEVER reach the terminal/PTY. |
| find chord → xterm/PTY | the Cmd+F/Ctrl+F chord is intercepted main-side (Plan 01) and must never reach the focused xterm. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-03 | Tampering (wrong-target input) | SearchBar `<input>` | mitigate | The bar is a DOM sibling overlay (NOT inside `.xterm`); its `onKeyDown` calls `stopPropagation()` so chars/Enter/Esc go to the input, never the PTY (Pitfall 3). The existing `attachCustomKeyEventHandler` focus gate (returns false when focus is not on `.xterm-helper-textarea`) composes cleanly. When the bar is closed it is unmounted, so it cannot intercept any keystroke (SC3). |
| T-07-04 | Tampering | search-over-PTY injection | accept (N/A) | `SearchAddon` decorations are DOM/canvas overlays; search runs against the in-memory xterm buffer and is NEVER written into the PTY (consistent with `handleClear` never injecting `clear`/Ctrl+L). No untrusted input crosses IPC — the search query stays renderer-local (explicitly-bounded surface). |
| T-07-05 | Denial of service | dangling addon/decoration on session removal | mitigate | `searchRef.current?.dispose()` + the `onDidChangeResults` unsubscribe run in the SessionView cleanup before `term.dispose()` (Pitfall 4) — no leaked subscription. |
| T-07-02 | Elevation | bridge surface | mitigate | This plan adds ZERO bridge keys (the find chord rides the existing 'session:switch' channel; the surface count stays at the 20 frozen by Plan 01). No `security.guard` change. |
</threat_model>
