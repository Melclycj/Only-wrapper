    The current bug: `handleToggleCase` calls `runSearch(query, { caseSensitive: next, decorations })` and `runSearch` calls `addon.findNext`, which moves to the NEXT match. Use the SearchAddon API to re-highlight without advancing — call `addon.clearDecorations()` then re-issue the search with the flipped `caseSensitive` flag and `decorations` (so `onDidChangeResults` re-fires the new count) but in a way that does NOT advance past the current active match. Concretely: re-run `findNext` from BEFORE the current active position rather than after it — i.e. clear the active selection first (`addon.clearActiveDecoration()` / clear selection) so the re-issued find re-evaluates the buffer and lands on the FIRST match at-or-after the top of the buffer for the new case mode, OR (preferred, simplest deterministic) gate the toggle through `decideCaseToggle` and, when `shouldRecompute`, re-issue the search with `incremental: true` in the ISearchOptions so it EXPANDS the current selection in place rather than stepping to the next match (the xterm `incremental` flag is documented as "expand the selection if it still matches … affects findNext, not findPrevious"). Verify on the live canvas (Task 3) that the active index does NOT increment on repeated Aa clicks and the count reflects the new case mode.
