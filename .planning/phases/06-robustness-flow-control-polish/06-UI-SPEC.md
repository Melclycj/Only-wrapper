# Phase 6 — UI Design Contract

> Visual and interaction contract for Phase 6: Robustness + Flow-Control Polish. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **This phase extends existing surfaces — it introduces NO new design system.** All tokens are pulled verbatim from `.planning/DESIGN.md` (the warm "parlour" system) and the already-shipped `:root` token set in `src/renderer/terminal.css`. The shadcn gate is **not applicable**: the project is mid-build (phase 6 of 9) on a locked, hand-rolled CSS custom-property design system; introducing shadcn/Tailwind here would contradict DESIGN.md's locked visual authority and the four prior shipped phases.
>
> **New / extended surfaces this phase owns:**
> 1. **Agent-state presentation layer** (SC4 / TERM-09, D-06..D-10) — the status dot/badge gains an attention overlay (amber "Waiting for you" / blue "In progress" / slate "Free") over the 5 process statuses, shown on every sidebar row + the identity header.
> 2. **Spawn / cwd error surfacing** (SC2, D-01..D-05) — red `error` badge + tooltip AND an error card (reusing `IdleCard`) with Edit + Retry recovery.
> 3. **Session header control cluster** (SC5 / TERM-12, D-11..D-13) — Clear (always) · Restart (when running) · Start ▶ (when not running) in `IdentityHeader`, plus a Clear keyboard chord.
> 4. **"Start without command"** secondary menu item (D-14) + **"Browse…" folder picker** (folded todo).
> 5. **Alt-screen reset** on restart / abnormal exit (SC3, D-15) — a frame-cleanliness behavior on the kept-alive xterm (no visible chrome).
>
> Pre-locked decisions D-01..D-16 are LOCKED contract inputs — formalized here, not relitigated. SC1 (backpressure watermark) is validation-only and has no new UI surface.

## Typography

**Canonical scale for new surfaces: 4 sizes, 2 weights.** All Nunito except the mono role (JetBrains Mono). Inherited Phase-4/5 values reused unchanged are documented in the exceptions block so they read as intentional continuity.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label / caption (agent-state badge label, error-card helper line) | 12px | 700 (bold) for labels · 400 (regular) for captions | 1.3–1.5 |
| Body / control / mono (error-card values, header control labels, body copy) | 13px | 400 (regular) · 700 on action button labels | 1.5 |
| Row / input | 14px | 400 (regular) | 1.4 |
| Heading (error-card session name) | 20px | 700 (bold) | 1.2 |

**Canonical sizes: 12 / 13 / 14 / 20 px. Canonical weights: 400 (regular) + 700 (bold).** Every weight used by a new surface is one of these two; every size is one of these four.

**Inherited typography exceptions (Phase 4/5 shipped — reused as-is, DO NOT re-baseline):**
- `.status-badge` **11px / 700** — the agent-state badge reuses the shipped `.status-badge` component verbatim; only its `--accent` + label text change. No new size introduced.
- `.identity-header .row-name` **14px / 600** (shipped) — the header name is unchanged by this phase; controls sit to its right.
- `.idle-card-name` **20px / 700**, `.idle-card-label` **12px / 700**, `.idle-card-value` **JetBrains Mono 13px / 400** — the error card reuses these shipped idle-card type roles verbatim.

Notes:
- The agent-state label ("Waiting for you" / "In progress" / "Free") renders in the SAME `.status-badge` 11px/700 chip used for process statuses — agent-state is a *presentation overlay*, never a visually distinct new badge component (D-06).
- Error-card values (cwd path, OS reason) render in **JetBrains Mono 13px / 400** so the failing path reads as a literal (reuses `.idle-card-value`).

## Color

The full palette is `:root` in `terminal.css` (DESIGN.md subset). The agent-state ramps come from DESIGN.md §"Status system"; oklch values are authoritative. 60/30/10 split maps the parlour system.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--surface` `#ffffff` + `--bg-sunk` `oklch(0.955 0.01 85)` | Identity-header surface, error-card body, recessed inset config block. |
| Secondary (30%) | `--term-bg` `#1e232c` (viewport gutter) + `--ink`/`--ink-soft`/`--ink-faint` ink ramp + `--line`/`--line-soft` borders | Terminal-area background behind the error card; all text + hairlines structuring the warm chrome; neutral resting state of header control buttons (`--ink-soft`). |
| Accent (10%) | agent-state + status accents (see reserved list) | Reserved list below. |
| Destructive | `oklch(0.58 0.16 25)` (red ramp) | The `error` status badge/dot + tooltip + error-card inline line + Close control hover (existing). No NEW destructive action introduced this phase (Close stays in the context menu, unchanged from Phase 3 D-03a). |

**Agent-state accent ramps (DESIGN.md §"Status system", D-07 — authoritative oklch):**

| Agent-state | Label | Accent (`strong`) | When |
|-------------|-------|-------------------|------|
| `in-progress` | **"In progress"** | blue `oklch(0.62 0.14 248)` | `running` AND output actively flowing |
| `waiting` | **"Waiting for you"** | **amber `oklch(0.66 0.15 60)`** | `running` AND idle ≥ threshold AND last line matches the prompt set (= TERM-09 / SC4) |
| `free` | **"Free"** | slate `oklch(0.64 0.02 260)` | `running` AND idle, no prompt-pattern match |

Process-status accents are unchanged from `status-colors.ts`: `exited`→green `oklch(0.60 0.13 150)` "Done/Finished"; `not_started`/`stopped`→slate `oklch(0.64 0.02 260)`; `error`→red `oklch(0.58 0.16 25)`. The agent-state ONLY overrides presentation while the process status is `running` (D-07).

Accent reserved for (explicit — never "all interactive elements"):
- **Status dot + status-badge** on each sidebar row, in the identity header, and on the collapsed-rail dot — now carrying the agent-state accent/label when the session is `running` (the ONLY new accent usage this phase).
- **Amber `oklch(0.66 0.15 60)` is reserved exclusively for the "Waiting for you" agent-state** — it is the highest-attention signal (rank 0) and appears nowhere else in the chrome.
- **Start (▶) header/row control accent on hover/focus** — blue `oklch(0.62 0.14 248)` (existing `.row-control-start` pattern, reused in the header cluster) to telegraph "this starts the session".
- **Clear / Restart header controls** — neutral `--ink-soft`, warming to `--ink`/`--bg-sunk` on hover (existing `.row-control` pattern); they do NOT take an accent color (they are not the primary "go" affordance and not destructive).
- **Input/dropdown + focusable-button focus ring** `oklch(0.62 0.14 248)` (blue) — the single, consistent focus affordance color (already shipped); header control buttons and the "Browse…" button inherit it.

**Best-effort honesty (D-10):** the agent-state is the session's *real* computed state, shown on ALL running sessions (sidebar rows + header). Amber clears automatically when the session leaves the waiting condition (new output → blue; quiet-no-prompt → slate). It is never an acknowledge-on-view notification and never a separate badge — it never "nags" because it is honest state.

## Copywriting Contract

Tone: warm, first-person-friendly, parlour-not-dev-tool (DESIGN.md aesthetic). Sentence case, no exclamation-spam, no jargon. Error messages name the problem AND the fix.

| Element | Copy |
|---------|------|
| Agent-state label — waiting (D-07/TERM-09) | **"Waiting for you"** (badge label; amber). The conservative best-effort signal — never "Needs attention!!!" or alarmist. |
| Agent-state label — in progress | **"In progress"** (badge label; blue) |
| Agent-state label — free | **"Free"** (badge label; slate) |
| Spawn error — missing cwd (D-05, specific) | **"Working directory not found: {path}"** (the configured path is shown verbatim, mono) |
| Spawn error — other failure (D-05, generic) | **"Couldn't start session: {os reason}"** (the OS error message, sanitized of control chars per WR-04) |
| Error-card body heading (D-03) | session **name** + status badge reading **"Error"** (red) — identity IS the heading; below it the specific error message |
| Error-card helper / recovery hint (D-04) | **"Check the working directory and shell, then fix them or try again."** (`--ink-faint`, 12px/400) |
| Error-card Edit action (D-04) | button label **"Edit"** — opens the existing edit modal to fix cwd/shell |
| Error-card Retry action (D-04) | button label **"Retry"** — re-attempts the spawn (the primary, blue "go" affordance in the card) |
| Sidebar error tooltip (D-03) | the same error message as the card (`title=` attribute on the error row) |
| Header control — Clear (D-11) | button label **"Clear"** (always present). `aria-label="Clear terminal"`. |
| Header control — Restart (D-11) | glyph **↻** + visually-hidden / `aria-label="Restart session"` (shown when `running`) |
| Header control — Start (D-11) | glyph **▶** + `aria-label="Start session"` (shown when not running) |
| Context-menu — Start without command (D-14) | item label **"Start without command"** (secondary; the primary Start ▶ runs the saved startup command) |
| Folder picker button (folded todo) | button label **"Browse…"** beside the working-directory field; opens the native OS directory picker |
| Destructive confirmation | **None new this phase.** The existing destructive Close confirm (Phase 3 D-03a) is unchanged. Clear, Restart, Start, "Start without command", Retry, and Browse are all non-destructive — no confirm. (Clear drops scrollback only, not the process — the prompt line is preserved, iTerm/VSCode Cmd+K semantics; no confirm needed.) |

## Interaction Contracts

> The surfaces this phase introduces or extends, spec'd prescriptively for the executor.

### 1. Agent-state presentation layer (SC4 / TERM-09 — D-06..D-10)

The status dot/badge gains an attention overlay. **No new badge component, no 6th process status** — the shipped `.status-badge` chip simply receives a different `--accent` + label when a `running` session is classified.

- **Resolver:** a `presentation(status, agentState)` function in `status-colors.ts` returns `{ label, accent }`. When `status === 'running'` and an `agentState` is present, it returns the agent-state ramp; otherwise it returns the existing `STATUS_STYLE[status]`. (See 06-RESEARCH.md §"status-colors mapping".)
- **Where shown:** every sidebar row's `.status-badge` + `.status-dot`, the collapsed-rail `.collapsed-status-dot`, and the `IdentityHeader` badge — all consume the resolved presentation. Computed for ALL running sessions (D-10), so a backgrounded session's amber dot is visible in the rail while you work in another session.
- **The three running sub-states (D-07):**
  - **blue "In progress"** — output is actively flowing (a chunk arrived within the idle threshold).
  - **amber "Waiting for you"** `oklch(0.66 0.15 60)` — output has been quiet for the idle threshold AND the last non-empty line matches the prompt set. This is the TERM-09 / SC4 signal.
  - **slate "Free"** — quiet, no prompt-pattern match.
- **Detection (RESEARCH-recommended, renderer-side, zero IPC):** computed in `SessionView` off the `onPtyData` stream it already consumes; idle threshold **`IDLE_MS = 800`** (Claude's discretion, D-08 — tune in E2E); conservative trailing-line regex (D-09) tested against the last non-empty line after ANSI strip. This is heuristic/best-effort by contract — false "waiting" alarms are worse than misses (CONTEXT §Specifics), so the gate is idle-AND-pattern.
- **Clearing (D-10):** amber clears automatically — new output flips to "In progress"; quiet-without-prompt flips to "Free". Merely viewing the session does NOT clear it (it is honest state, not a notification).
- **Visual treatment:** identical chip geometry to the process-status badge (8px dot, 11px/700 label, `gap 5px`); only color + text differ. No animation, no pulse, no count badge — calm by design.

### 2. Spawn / cwd error surfacing (SC2 — D-01..D-05)

A failed spawn (missing cwd or any spawn failure) must surface in BOTH places (D-03) and offer recovery (D-04).

- **Sidebar (D-03):** the row status flips to **`error`** — red dot/badge `oklch(0.58 0.16 25)` "Error" (existing `STATUS_STYLE.error`). The full message rides the row as a `title=` tooltip. No new row layout.
- **Error card (D-03):** reuse `IdleCard` for the `error` branch (it already has an `error` state, lines 88–93). The card stays visible in `.terminal-area` instead of a confusing blank/dead terminal. It shows: identity region (icon → name → red "Error" badge), the specific message (mono path for the cwd case), the helper line, then the action row.
- **Message (D-05):** missing cwd → **"Working directory not found: {path}"** (the user-supplied path verbatim, JetBrains Mono); any other failure → **"Couldn't start session: {os reason}"**. The message is sanitized of control characters before render (WR-04). Transport rides the existing `onPtyStatus` `PtyStatusPayload.notice` field — **zero new bridge keys**.
- **Recovery action row (D-04):** two buttons at the card bottom, gap 8px, right-or-stretch aligned:
  - **"Edit"** — neutral button (`--ink-soft`, `1px solid var(--line)`, warms on hover) → opens the existing `SessionEditModal` to fix cwd/shell.
  - **"Retry"** — the primary blue "go" affordance (reuse the `.idle-start-button` blue-outline → blue-fill-on-hover treatment) → re-attempts the spawn via the existing Start path.
  - Both `data-testid` (`error-card-edit`, `error-card-retry`); both Tab-focusable with the blue focus ring.
- **No silent home fallback (D-02):** a user-specified-but-missing cwd is an error — it must NOT silently spawn in `~`. (Behavioral contract; not a visible surface, but it is the reason the error card exists.)

### 3. Session header control cluster (SC5 / TERM-12 — D-11..D-13)

`IdentityHeader.tsx` is identity-only today (Phase 4 D-05). It gains a right-aligned control cluster.

- **Layout:** the existing `.identity-header` (`display: flex; gap: 8px; padding: 8px 14px`) keeps icon → name (`flex: 1 1 auto`) → status badge, then a NEW right-aligned `.header-controls` cluster (`display: inline-flex; gap: 4px; margin-left: auto` — or after the badge). The cluster does not crowd the badge.
- **Controls (contextual, D-11):**
  - **Clear** (always) — a text-labelled button "Clear" (`8px 16px` pad, radius 10px, 13px/700, neutral `--ink-soft` → warms on hover) OR a ⌫-glyph 24×24 `.row-control`-style icon button (executor's choice; prefer the labelled button in the header for discoverability since space allows). `data-testid="clear-terminal"`, `aria-label="Clear terminal"`.
  - **Restart** (when `status === 'running'`) — ↻ glyph icon button (reuse the shipped `.row-control` 24×24 neutral treatment). `data-testid="header-restart"`, `aria-label="Restart session"`.
  - **Start ▶** (when not running — `not_started`/`stopped`/`exited`/`error`) — ▶ glyph icon button with the **blue** `.row-control-start` hover/focus accent (the primary "go" affordance). `data-testid="header-start"`, `aria-label="Start session"`. This fixes the folded Start-discoverability todo.
- **No new Stop button (D-11):** the destructive **Close** stays in the right-click context menu only. There is NO non-destructive Stop button (Phase-3 D-03a preserved).
- **Keyboard (D-13):** all controls are native Tab-focusable `<button>`s (Enter/Space activates → satisfies SC5 "keyboard-accessible"), each with the consistent 2px blue `:focus-visible` outline. PLUS a global **Clear chord**: **Cmd+K (macOS) / Ctrl+Shift+K (Windows)** — intercepted main-side via the Phase-4 "app-wins" `before-input-event` seam so it never reaches xterm/PTY (works inside vim/tmux). Restart/Start stay button-only (no global chord). Ctrl+K is deliberately avoided on Windows (readline kill-line).
- **Clear behavior (D-12):** Clear = client-side `term.clear()` on the active session's kept-alive xterm (drops scrollback, **preserves the current prompt line** — iTerm/VSCode Cmd+K semantics). It does NOT inject `clear`/Ctrl+L into the PTY (no shell-history pollution, consistent across shells).

### 4. "Start without command" + folder picker (D-14 + folded todo)

- **Start without command (D-14):** a SECONDARY item in the row right-click `ContextMenu` (reuse `.context-menu-item`), label **"Start without command"**, shown for a startable row that has a saved `startupCommand`. The primary Start ▶ (row control + header + menu "Start") runs the saved startup command; this item spawns a bare shell in the saved cwd/shell, skipping the TERM-05 auto-run for that one launch. Non-destructive, no confirm.
- **Folder picker (folded todo):** a **"Browse…"** button beside the working-directory field in `SessionEditModal.tsx`. Styled as a neutral secondary button matching `.edit-input` height (`8px 10px` pad, radius 10px, `1px solid var(--line)`, `--ink-soft`, Nunito 14px) sitting inline-end of the cwd field. Click → `pickDirectory()` IPC → native `dialog.showOpenDialog({ properties: ['openDirectory'] })` → fills the cwd field with the absolute path. Cancel returns null → field unchanged. The existing CR-01 validator still gates the value. `data-testid="browse-cwd"`. (This is the one intentional new bridge key this phase — `pickDirectory`; the agent-state + error message add NO new keys.)

### 5. Alt-screen reset (SC3 — D-15) — behavior, no visible chrome

A kept-alive xterm can be stuck in the alternate-screen buffer if vim/less was killed without a normal exit. This phase resets the *displayed frame* so reopening shows a clean prompt — it has no new visible control, only correct terminal-frame behavior.

- **On restart** (the second `'running'` transition in `SessionView`): exit the alt-screen with `\x1b[?1049l` (preserves primary-screen scrollback — honors Phase-3 D-03 "preserve scrollback across restart"), THEN write the existing `— restarted HH:MM —` separator. Reset-then-separator ordering matters.
- **On abnormal exit** (exited/error, not user-initiated, surfaced in `onPtyExit`): the frame is genuinely dead → `term.reset()` (full RIS) then write the exit notice, so a killed-vim frame never freezes on screen.
- **The reset-vs-scrollback tension (RESEARCH Open Q1)** is a human-verify checkpoint item: the surgical `\x1b[?1049l` on restart vs full `reset()` on abnormal exit must be confirmed in E2E (kill vim, restart, inspect frame + scrollback). No new copy or control — purely a terminal-fidelity behavior.

### 6. Backpressure watermark (SC1 — D-16) — validation only, no new surface

The renderer HIGH/LOW watermark (`flow-control.ts` + `SessionView.tsx`, FLOW_HIGH=100000 / FLOW_LOW=10000) already exists (Phase 2-04). This phase validates it under `cat /dev/urandom | head -c 100M` (no freeze / crash / drop; visible pause+resume) and deletes the dead `TerminalPane.tsx` to resolve the flow-control duplication. **No UI surface, no copy, no control.** Listed here only to record that SC1 has no design contract beyond "the terminal stays responsive and lossless".
