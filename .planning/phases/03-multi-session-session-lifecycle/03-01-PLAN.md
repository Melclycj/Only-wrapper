Build the producer side of the multi-session lifecycle: extend the main-process `PtyManager` from a single PTY to a per-session record with a 5-state status machine, graceful platform-aware stop, identity-preserving restart, and visible startup-command injection — plus the typed contextBridge surface (`ptyStop`, `ptyRestart`, `onPtyStatus`, `listSessions`) the renderer will consume in plan 03-02/03-03. <threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| renderer → main (contextBridge) | Untrusted renderer args (`id`, `startupCommand`, cols/rows) cross into main where node-pty lives. The 4 new bridge methods (`ptyStop`, `ptyRestart`, `onPtyStatus`, `listSessions`) widen this boundary. |
| main → child process | Each PTY is a real OS child inheriting the full parent env; N sessions = N children to reap. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering / DoS | `ptyStop`/`ptyRestart` with a forged/unknown `id` to kill or hijack a session | mitigate | Validate `id` against the live session/record store before acting; unknown/forged ids are ignored (reuse the existing `sessions.get(id)` guard pattern). Unit-tested via the unknown-id path. |
| T-03-02 | Tampering | New IPC channels (`pty:stop`/`pty:restart`/`pty:list`/`pty:status`) registered non-idempotently or with asymmetric teardown → duplicate handlers fire N times | mitigate | Register inside the `ipcRegistered` guard; add symmetric `removeHandler`/`removeAllListeners` in `unregisterIpc`. Enforced by `ipc-registration.test.ts` (must stay GREEN). |
| T-03-03 | Elevation / Injection | `startupCommand` treated as app-evaluated shell input | accept | The app writes it as raw keystroke bytes into the user's OWN shell — no app-side `eval`/`exec`; it runs with exactly the user's privileges, no new trust boundary crossed (the user configured it for their own session). |
| T-03-04 | Information Disclosure | `startupCommand` (may contain a token) or raw PTY bytes logged | mitigate | Log lifecycle only ("startup command injected", spawn pid, exit code) — never the command text or PTY output (Security V7, carried from Phase 2). |
| T-03-05 | DoS / resource leak | N orphaned child processes + an in-flight SIGKILL grace timer surviving quit | mitigate | `disposeAll()` kills every live pty AND `clearTimeout`s any pending `killTimer` on window-close/before-quit. |
| T-03-06 | Spoofing (surface drift) | An unreviewed bridge key (e.g. raw `ipcRenderer`) leaks through the widened surface | mitigate | `EXPECTED_API_KEYS` updated in lockstep with the 4 new methods; `security.guard.test.ts` asserts the EXACT 12-key set and `not.toHaveProperty('ipcRenderer')`. |
| T-03-SC | Tampering | npm/pip/cargo installs | accept | No new packages this phase (RESEARCH §Package Legitimacy Audit — all deps unchanged from Phases 1-2); no install task, so no slopcheck checkpoint required. |

No HIGH-severity threat is left without a mitigation.
