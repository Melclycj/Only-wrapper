# Phase 06.1: Terminal lifecycle state machine + agent-state detection redesign - Context

**Gathered:** 2026-06-07
**Status:** Ready for planning

<domain>
## Phase Boundary

Redefine the session/terminal lifecycle and fix the issues that Phase 6's
human-verify gate exposed in hands-on testing. Two axes:

- **Axis 1 — Process-status state machine + Start surface.** A simpler, lower-step
  lifecycle: a **Working Area** of live sessions + an **Inactive List** of dormant
  configured sessions. No "Stop" verb. Start is always discoverable. Fix the
  restart/scrollback/restore bugs.
- **Axis 2 — Agent-state detection redesign.** Replace Phase 6's output-silence
  idle detection (which left `claude --rc` permanently blue) with the
  **frame-stability** model validated by spikes 001/002.

Also in scope: the **terminal-fidelity regression** (broken scroll + `[%30/]`
escape garbling after alt-screen apps) — a Core Value break.

**This phase SUPERSEDES the unsigned TERM-09 (agent-state) and TERM-12
(header-controls) work from Phase 6 (plans 06-03 / 06-04).** Phase 6's 06-01
(bridge/probe fixes, dead-code removal) and 06-02 (SC2 spawn-error handling) are
sound and **retained**. `06-VALIDATION.md` stays `nyquist_compliant: false` —
Phase 6 is not signed off; this phase finishes the lifecycle + TERM-09/TERM-12 work.

**Requirements:** TERM-09 (waiting-for-input signal), TERM-12 (header clear/restart
controls) — re-delivered here on a corrected model.

**Explicitly NOT in this phase:**
- **Real agent conversation resume** (claude `--resume` carrying prior context) —
  out of app scope project-wide. The app never replays terminal history; Start is
  always a fresh process.
- **Windows-specific behavior** — stays behind the existing seams (Phase 8).
- **Terminal search (Ctrl+F) + scrollback-size config** — Phase 7 (TERM-10/11).
- **Reading agent/page content** to detect state — privacy line; detection uses
  rendered-frame *shape/stability* only, never conversation content.

<decisions>
## Implementation Decisions

### Session lifecycle & states (Axis 1)

- **D-01: Two-bucket model — no "Stop" verb.** The user-facing lifecycle is
  **Working Area** (live processes only) + **Inactive List** (dormant, *configured*
  sessions only). There is no in-place "stopped" limbo and no Stop button. This
  **supersedes** Phase-6 D-11's running/stopped header framing and reframes Phase-3's
  5-state *process* model into a two-bucket *user* model (the PTY may still transition
  internally; the UI presents two buckets). North star: minimize steps for simple
  tasks (e.g. starting a terminal).
- **D-02: Ephemeral vs configured, with auto-promotion.** A session with **no stored
  metadata** (no name / folder / cmd / icon) is **ephemeral** — throwaway: Close → gone,
  and it is **never persisted across app restart**. A session **auto-becomes
  "configured" the instant ANY metadata is set** (name/folder/cmd/icon). `+ New`
  creates an ephemeral plain-shell session (the low-friction quick terminal).
- **D-03: Configured-session lifecycle.** Configured sessions persist across app
  restart. **Remove** → moves to the Inactive List (process killed, recipe kept).
  **Delete** (from the Inactive List) → permanent. While live, ephemeral and
  configured sessions look identical in the Working Area; the distinction only
  manifests at removal/persistence.
- **D-04: Start = fresh process, NO history.** Starting a session (Inactive List →
  Working Area, or `+ New`) spawns a **fresh** process with **no scrollback/history
  restore** — clean screen, then re-run the saved startup cmd (TERM-05). The app
  **never replays old output**; real context continuity is `claude --resume`'s job
  (out of scope). *(Chose "recipe only"; rejected a scrollback-snapshot/Option-C.)*
- **D-05: Exit vs spawn-failure are different paths.** A configured session whose
  process **exits on its own** (`exit` / Ctrl-D / crash) → **drops straight to the
  Inactive List** (Working Area stays live-only). A session that **fails to spawn**
  (e.g. missing cwd — SC2) → **stays in the Working Area** showing the error card with
  **Edit / Retry** (Phase-6 D-03/D-04, retained) so the user can see and fix it; it
  does NOT auto-drop to inactive. An **ephemeral** session that exits → just gone.
- **D-06: Control placement (fixes "no Start UI anywhere").** A **live** session's
  header shows **Clear + Restart** (no Start, no Stop) plus a way to **Remove**.
  **Start ▶** lives on **every Inactive List entry** (alongside Delete and a
  "Start without command" option). Start is discoverable because it lives exactly
  where non-running sessions live. This **supersedes** Phase-6 D-11 (contextual
  Start-in-header-when-not-running).

### Restart & restore (Axis 1)

- **D-07: In-place Restart preserves scrollback.** Restart (a live-session header
  action) = kill + respawn the **same** session in place: **preserve** the existing
  on-screen scrollback, exit the alt-screen (`\x1b[?1049l`) FIRST, then write the
  `— restarted HH:MM —` separator. **Fix the bug where repeated Restart trims
  scrollback** — it must not trim.
- **D-08: App-restart restore = Inactive List, no separator.** On app restart,
  previously-live configured sessions return in the **Inactive List** (their OS
  processes died with the app). A **first Start of a restored/dormant session shows
  NO `— restarted —` separator** — that separator is ONLY for in-place restart of a
  *live* session. This **fixes the "fresh terminal prints `— restarted —`" bug**.

### Agent-state detection (Axis 2)

- **D-09: Detection = frame-stability (replaces output-silence).** Tick the **live
  xterm `term.buffer.active` viewport** (~100ms), FNV-1a hash the visible text;
  unchanged for **T ≈ 400–600ms** ⇒ *settled*, else *working*. The animated
  "Thinking…" line churns the viewport text ⇒ correctly *working* — the property
  Phase 6's byte-silence model lacked. No headless and no IPC in production (the
  app already holds the live xterm). **Supersedes Phase-6 D-07/D-08/D-09.** Spikes
  001/002 are the reference (`record.cjs` is the algorithm).
- **D-10: Classify the cursor region of the settled frame.** WAITING signals =
  **confirmation footer** ("Esc to cancel · Tab to amend · ctrl+e"), **numbered menu**
  (≥2 "N." lines), `[y/N]`, **trailing `?`**, **password prompt**. A **shell prompt on
  the active line is authoritative FREE**. **`❯` (arrow_marker) is NOT a signal** —
  real Claude Code shows it persistently in its input caret (false-fired 10/11 settles
  in the 002 capture). Keep regexes linear/anchored (ReDoS-safe) on a bounded tail.
- **D-11: Three agent-states; amber = "blocked, needs a decision".** blue **In
  progress** (working) / amber **Waiting for you** (settled on a confirmation prompt
  ONLY) / slate **Free** (settled, no prompt). An agent that **finished and is idle at
  its input box reads as Free** — amber is reserved for the high-signal blocked case.
  **No fourth "your turn" tier** (would require brittle agent-chrome recognition).
- **D-12: Overlay scope (carries Phase-6 D-10).** The overlay applies ONLY to live
  (running) sessions; it is computed for ALL live sessions and shown on the sidebar
  row + header; it **clears state-driven, never on view** (honest state, not a
  notification — never nags). Inactive sessions have no agent-state.

### Terminal fidelity & approach (Axis 4)

- **D-13: Scroll / `[%30/]` regression is an in-scope must-fix (Core Value).**
  Working hypothesis (for the planner's debug pass, not a locked cause):
  **mouse-tracking mode left enabled** — an alt-screen app (vim/claude) enables mouse
  reporting (`\x1b[?1000h`/`?1006h`) and exits without disabling it, so scroll-wheel
  events get sent to the shell as escape sequences (echoed as `[%30/]`-style garbage)
  instead of scrolling xterm's buffer. Phase 6's new alt-screen writes
  (`\x1b[?1049l` / `term.reset()`) are the prime suspect for causing or failing to
  clean this up. Required end state: **scroll-wheel scrolls the xterm buffer; mouse
  mode engages only when the running app requests it and is reset on exit.** Root-cause
  via a debug pass during planning/execution.
- **D-14: Refactor 06-03 / 06-04 in place (not revert + rebuild).** Keep the reusable
  infrastructure — `status-colors.ts` ramps + `presentation()` resolver, the
  `SessionView` detector seam, the header control markup, the Clear chord + bridge —
  and surgically swap what changed: detection → frame-stability (D-09/D-10), controls →
  the new no-Stop / Inactive-List lifecycle (D-01/D-06), and **re-derive the alt-screen
  writes** (the D-13 regression suspect). Additive git history.

### Carried forward (locked — do NOT re-open)
- SC2 spawn/cwd error handling, error card + Edit/Retry, no-silent-home
  (Phase-6 D-01…D-05) — built in 06-02, retained.
- **Clear = client-side `term.clear()`**, no PTY injection (Phase-6 D-12).
- **Clear chord Cmd+K (mac) / Ctrl+Shift+K (win)** on the existing `session:switch`
  channel — no new bridge key (Phase-6 D-13).
- `pickDirectory` / **Browse…** folder picker; **"Start without command"** bare-shell
  launch (Phase-6 D-14) — now an Inactive-List-entry / context-menu option.
- **Controls must be genuinely keyboard-focusable** — fix the bug where Tab/Space were
  swallowed by the terminal instead of reaching the header controls (focus management).
- Identity invariants: error/restart/restore must never change `logicalId` or conflate
  it with `ptyPid`.

<specifics>
## Specific Ideas

- **North star: fewer steps for simple tasks.** Starting a terminal should be
  near-zero ceremony (`+ New` = instant ephemeral shell); only sessions worth keeping
  (given metadata) earn persistence.
- **Canonical scenario (unchanged):** `🛋️ Parlour Claude RC` — while you work in
  another session, a backgrounded `claude --rc` that hits a `[y/N]` / `❯`-menu
  confirmation goes **amber "Waiting for you"** within ~1s (frame settles on the
  prompt), and clears itself when it moves on. Validated achievable by spike 002.
- **Real-data lesson:** the `❯` caret is ambient in Claude Code — detection keys on the
  confirmation **footer/menu**, never `❯`.
- **"Stopped session = a recipe, not a frozen process"** — the app stores config and
  re-creates fresh; it never freezes/thaws a live process or replays history.
