# Phase 06.1 Plan 02: Renderer SEAM A (frame-stability) + SEAM B (mouse/alt-screen fidelity) Summary

**SessionView's output-silence agent detector is replaced by a frame-stability tick over the live `term.buffer.active` (D-09), and every process death/restart now emits a full mouse-mode reset so the scroll-wheel scrolls the xterm buffer after a TUI dies (D-13) while scrollback survives ≥3 restarts and `term.reset()` is gone (D-07).**

## Accomplishments

- **SEAM A (Task 1):** Removed the temporary Plan-01 bridge (local `IDLE_MS`/`classifyIdle` shim) and the byte-tail output-silence detector (`agentTail`/`idleTimer`). Imported `classify`/`TICK_MS`/`SETTLE_MS`. Added module-local `viewportLines()` + `fnv1a()` ported verbatim from the spike `record.cjs` (reads only `term.buffer.active` `viewportY..+rows`, never scrollback). Replaced the idle timer with a `setInterval` tick: hash the viewport, emit `'in-progress'` on change, `classify(lines)` on a ≥`SETTLE_MS` settle — gated on `agentRunning`, change-only emit, armed once per `id`, ticking even on hidden panes (D-12). On leave-running it resets `lastHash`/`changeAt` (not clear the interval — the same term persists across in-place restart); the interval is cleared in the effect cleanup.
- **SEAM B (Task 2):** Added `MOUSE_RESET` (every DECRST mouse-tracking + encoding mode) and `ALT_SCREEN_EXIT` (`\x1b[?1049l`) constants. `onPtyExit` emits `MOUSE_RESET` on every death (kill OR restart-stop). The `running` transition emits `MOUSE_RESET` unconditionally + `ALT_SCREEN_EXIT` only when `buffer.active.type==='alternate'` (D-07 — no needless toggle/trim of a plain shell). Abnormal-exit replaced `term.reset()` with `MOUSE_RESET + ALT_SCREEN_EXIT` (scrollback-preserving). No `term.reset()` remains in SessionView.
- **Smoke (Task 2):** Added `mouseTrackingModeOf`, `bufferTypeOf`, `viewportYOf`, `scrollViewportUp` helpers. Extended `alt-screen-reset.smoke.test.ts` to 4 specs covering: abnormal-exit → normal buffer + `mouseTrackingMode==='none'` (D-13); ≥3-restart `KEEP_` scrollback survival asserted every iteration (D-07); restart resets `mouseTrackingMode` to `'none'` (D-13); scroll-wheel scrolls the buffer after a killed alt-screen+mouse TUI with no SGR garble (D-13). **All 4 pass** against the packaged app.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] MOUSE_RESET must fire on `onPtyExit`, not be gated on `hasRunBeforeRef` in the `running` branch**
- **Found during:** Task 2 (smoke RED on the two D-13 specs)
- **Issue:** The plan derived SEAM B from the existing restart seam, which gated the alt-screen write on `hasRunBeforeRef.current`. Empirically (proven by smoke), the initial spawn's `running` AND the **first** restart's `running` both broadcast BEFORE SessionView's status subscription binds — the same documented race the restart smokes already account for (the old separator test needed two restarts to see `— restarted`). Gating MOUSE_RESET on `hasRunBeforeRef` therefore skipped the user's first restart, leaving `mouseTrackingMode` hot (`drag`) and the wheel garbling — exactly the D-13 regression. Gating it on the `running` transition alone was unreliable.
- **Fix:** Emit `MOUSE_RESET` on `onPtyExit` (the ONE reliably-delivered death signal — `[process exited]` proved it always fires, on kill AND on every restart-stop) AND unconditionally on the `running` transition (idempotent no-op when the mode is already off). The `— restarted —` separator stays gated on `hasRunBeforeRef` so D-08 (no separator on a fresh dormant Start) is preserved.
- **Files modified:** src/renderer/SessionView.tsx
- **Verification:** the "restart resets mouse-tracking mode to none" smoke spec passes; manual diag confirmed `mouseTrackingMode: none` after a single restart with the rebuilt app.
- **Committed in:** ccbd86c

**2. [Rule 1 - Test correctness] Abnormal-exit spec assertion + scroll-wheel driver corrected for the scrollback-preserving model**
- **Found during:** Task 2
- **Issue (a):** With `term.reset()` replaced by the scrollback-preserving `MOUSE_RESET + ALT_SCREEN_EXIT`, the existing spec's `expect(buf).not.toContain(frame)` failed — the `frame` marker is part of the **echoed command line** the user typed into the primary buffer, which the scrollback-preserving exit (correctly) keeps. **Issue (b):** A synthetic `WheelEvent` does not drive xterm's viewport scroll under the WebGL renderer in this harness (verified across `.xterm-viewport`/`.xterm-screen`/pane), so the wheel spec's `viewportY` never changed.
- **Fix:** (a) the abnormal-exit spec now drives the alt-screen via a bare `\x1b[?1049h` (no in-text marker) and asserts `bufferTypeOf(id)==='alternate'` before the kill and `'normal'` + `mouseTrackingMode==='none'` after — the authoritative "the stuck alt frame did not survive" signal. (b) `scrollViewportUp` drives xterm's NATIVE viewport-scroll path (`vp.scrollTop` + a `scroll` event — exactly what a real wheel ultimately triggers; confirmed it moves `viewportY` 163→0), with the `mouseTrackingMode==='none'` assertion as the D-13 root-cause proof.
- **Files modified:** tests/smoke/alt-screen-reset.smoke.test.ts, tests/smoke/helpers/xterm-driver.ts
- **Verification:** all 4 smoke specs pass (8.2s) against the packaged app. The plan's Task 2 explicitly permitted updating the abnormal-exit assertion if it conflicted with the scrollback-preserving choice.
- **Committed in:** ccbd86c

## Issues Encountered

- **Stale-build red herring (the bulk of the time):** the WDIO smoke harness launches a PRE-PACKAGED binary at `out/Just-Wrapper-darwin-arm64/…` — it does NOT rebuild from source. Early smoke runs (and renderer-injected diagnostics) reflected the OLD bundle, making it look as if `onPtyExit`/`onPtyStatus` callbacks never fired and MOUSE_RESET had no effect. Running `npm run package` before each smoke run resolved it; with the rebuilt app the fixes behave exactly as designed. **Lesson for downstream waves: `npm run package` is mandatory before `wdio run` whenever renderer/main source changed.**
