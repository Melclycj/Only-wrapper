## Summary

This is a **surgical refactor of existing, working code** — every file named in CONTEXT.md was read in full this session, and the structures are sound. The detection redesign (Axis 2) is already spike-validated end-to-end; the planner does **not** re-derive it — it ports `record.cjs`'s tick/FNV-1a/classify loop from a headless emulator onto the live `term.buffer.active` xterm that `SessionView` already holds, replacing the `IDLE_MS` output-silence timer at lines 231–269 and 339–343 of `SessionView.tsx`.

Three concrete things genuinely need research and are answered below: (1) **D-13 root cause** — and it is determinable from the code: the app's `\x1b[?1049l` (alt-screen exit) does NOT reset mouse-tracking modes (1000/1002/1003/1006), so a killed/exited TUI that enabled mouse reporting leaves xterm capturing the scroll wheel as mouse-event escape bytes instead of scrolling its own buffer — the `[%30/]`-style garble. (2) **The lowdb ephemeral-vs-configured shape** — the Phase-5 store persists *every* `SessionRecord` in `listSessions()`; ephemerality is a new gate, cleanly added as a derived predicate (no schema-version bump strictly required, but a `version` bump + migration is the clean path). (3) **The restart scrollback-trim bug (D-07)** — repeated `term.reset()` on abnormal exit + `\x1b[?1049l` on restart interact with xterm's scrollback; the trim is almost certainly the alt-screen enter/exit pair discarding primary-buffer rows, not a bug in the separator write.

**Primary recommendation:** Re-derive the terminal-frame writes around a single new principle — *always emit a full mouse-mode reset (`\x1b[?1000l\x1b[?1002l\x1b[?1003l\x1b[?1006l`) plus `\x1b[?1049l` whenever a process dies or restarts*, never rely on `term.reset()` (it wipes scrollback → the D-07 trim) for the restart path. Port `record.cjs` verbatim as a renderer-side `useFrameStability` tick over `term.buffer.active`. Add an `ephemeral` derivation in main keyed on "has any metadata," and persist only configured records.

### Pattern 1: Port `record.cjs` onto the live xterm (D-09 — SEAM A)
**What:** Replace the `IDLE_MS` output-silence detector in `SessionView.tsx` (lines 231–269 + the reset at 339–343) with a `setInterval` tick that reads `term.buffer.active`, hashes the cursor-region viewport, and classifies on settle.
**The reference algorithm is `record.cjs` (read in full).** The production port differs only in *where the viewport comes from*: production reads the app's live `Terminal`, the spike reads a headless one. The exact buffer-read code carries over verbatim:

```ts
// Source: .planning/spikes/001-frame-stability-mechanism/record.cjs (viewportLines + fnv1a)
function viewportLines(term: Terminal): string[] {
  const b = term.buffer.active;
  const top = b.viewportY;          // first visible row index in the buffer
  const out: string[] = [];
  for (let i = 0; i < term.rows; i++) {
    const ln = b.getLine(top + i);
    out.push(ln ? ln.translateToString(true) : ''); // trimRight=true → ANSI-interpreted clean text
  }
  return out;
}
function fnv1a(str: string): string {
  let h = 0x811c9dc5;
  for (let i = 0; i < str.length; i++) { h ^= str.charCodeAt(i); h = Math.imul(h, 0x01000193); }
  return (h >>> 0).toString(16);
}
```
- `b.viewportY` = top visible row; loop `term.rows` rows = exactly the visible viewport (NOT scrollback). [VERIFIED: xterm.d.ts `IBuffer.viewportY`, `getLine`, `translateToString(trimRight)`]
- `translateToString(true)` returns trailing-trimmed, ANSI-interpreted text — a blinking cursor (render-layer) is naturally excluded; patterns run on clean text. (001 §approach)
- Tick: `TICK_MS = 100`; settle when the hash is unchanged for `T` (discretion 400–600ms). On change → `working` (`in-progress`); on settle → `classify()`.

### Pattern 2: The classifier — `classify()` from `record.cjs` (D-10)
The current `agent-state.ts` `classifyIdle()` encodes the WRONG model (output-silence + a `PROMPT_RE` that *includes* `❯`, line 49). **Rework it to the spike's `classify()`** — drop `❯` from the decision, window the cursor region (last ~4 non-empty lines), shell-prompt-active ⇒ FREE:

```ts
// Source: .planning/spikes/001-frame-stability-mechanism/record.cjs classify() — the 002-corrected form
function classify(lines: string[]): AgentState {
  const nonEmpty = lines.filter((l) => l.trim() !== '');
  const last = nonEmpty.length ? nonEmpty[nonEmpty.length - 1] : '';
  const region = nonEmpty.slice(-4).join('\n');                   // tight cursor-region window
  const shellPrompt = /[$%#]\s*$/.test(last) || /\w[^>]>\s*$/.test(last);
  const trailingQuestion = /\?\s*$/.test(last);
  const ynBracket = /\[y\/n\]|\(y\/n\)|\(yes\/no\)/i.test(region);
  const numberedMenu = (region.match(/^\s*[❯>]?\s*\d+\.\s+\S/gm) || []).length >= 2;
  const claudeFooter = /(esc to (cancel|interrupt)|tab to amend|ctrl\+e to|↑↓ to|enter to)/i.test(region);
  const passwordPrompt = /(password|passphrase).*:\s*$/i.test(last);
  // ❯ (arrow_marker) is DELIBERATELY ABSENT from the decision (002 kill-finding).
  if (shellPrompt && !trailingQuestion) return 'free';            // FREE(shell) — authoritative
  if (numberedMenu || ynBracket || trailingQuestion || passwordPrompt || claudeFooter) return 'waiting';
  return 'free';
}
```
All regexes are linear + anchored (`\s*$`, `^...$/gm`) → ReDoS-safe on a bounded viewport (the viewport is `rows × cols`, inherently bounded; no 4KB-tail concern remains because we read fixed-size frames, not a growing byte stream).

### Pattern 3: Tick lifecycle (mount / unmount / restart / hidden)
**What:** The tick must arm when the session is `running` and be torn down on unmount, gate-close, and restart.
- Arm the `setInterval` in the mount effect (keyed on `id`), gated on the existing `agentRunning` flag (set true on `running`, false on exit/stop — already wired at `SessionView.tsx` lines 320, 338).
- Clear the interval in: (a) the effect cleanup (line 367 region), (b) when `agentRunning` flips false (mirror the existing `idleTimer` clear at 339–343), and (c) it does NOT need re-arming on restart — the SAME `Terminal` instance persists across an in-place restart (keep-alive xterm), so the interval keeps running; just reset `lastHash`/`changeAt` so a restart's fresh frame is re-evaluated.
- **Hidden panes still tick correctly:** `term.write` runs unconditionally while hidden (lines 218, 248), so `buffer.active` stays current for a backgrounded session — this is REQUIRED for D-12 (a backgrounded `claude --rc` must go amber while you work elsewhere). Reading `buffer.active` does not require a WebGL context (the buffer is independent of the renderer), so detection works on the inactive, GPU-context-free panes too. [VERIFIED: SessionView.tsx lines 60–83, 218, 248]

### Pattern 4: Mouse-mode-safe frame reset (D-13 — SEAM B)
**What:** Whenever a process dies or restarts, emit an explicit mouse-mode disable BEFORE/INSTEAD OF relying on `1049l` or `reset()`. See the D-13 root-cause section for the full sequence. The principle: the app must defensively reset mouse tracking because a killed TUI never sends its own `rmcup`/mouse-disable.

### Pattern 5: Two-bucket bucketing is a pure render-time partition (D-01)
**What:** Working Area vs Inactive List is a partition of `sessions[]` by a predicate, NOT a new state machine. Live (has-or-had a PTY this app-session, status running) → Working Area; dormant (`not_started`, restored-from-store) → Inactive List. The existing `SessionManager` already computes `startedSessions` (line 501) and renders dormant rows via `IdleCard`; the planner extends this to a labeled two-section sidebar.

### Anti-Patterns to Avoid
- **`term.reset()` (RIS) on the restart path.** It wipes scrollback → directly causes the D-07 trim. Restart must use `\x1b[?1049l` + mouse-mode disable, never RIS.
- **Reading scrollback rows for detection.** `record.cjs` reads ONLY the viewport (`viewportY`..`viewportY+rows`). A broad scrollback scan re-introduces the 001 stale-menu false positive.
- **Using `❯` as a waiting signal.** The 002 kill-finding: 10/11 false positives.
- **Adding any IPC/bridge key.** Detection is renderer-local; `security.guard.test.ts` asserts exactly 19 keys (`EXPECTED_API_KEYS`). The ephemeral flag stays main-internal (not bridged).
- **Persisting agent-state or ephemeral-while-live as a status.** Agent-state is a presentation overlay (D-12); ephemeral is a persistence gate, not a 6th `SessionStatus`.

## Axis 4 (D-13) — Root-Cause Analysis: scroll / `[%30/]` mouse-mode regression

### Root cause (determinable from the code — HIGH confidence)

**The app never manages mouse-tracking modes, and `\x1b[?1049l` does not reset them.**

Evidence:
1. A full grep of `src/` for `1000`/`1002`/`1003`/`1006`/`mouse` found **zero** mouse-mode handling. The `Terminal` is constructed with no mouse options (`SessionView.tsx` lines 142–150). [VERIFIED: grep this session]
2. The only escape writes the app emits are `\x1b[?1049l` (restart seam, line 315) and `term.reset()` (abnormal-exit seam, line 332). [VERIFIED: SessionView.tsx]
3. xterm.js treats the alternate-screen buffer mode (1049) as **independent** of mouse-tracking modes (1000/1002/1003) and mouse encoding (1006). Exiting the alt-screen does NOT clear mouse tracking; only a full RIS (`term.reset()`) clears all DECSET modes. [CITED: xtermjs.org/docs/api/vtfeatures — "modes changing the active terminal buffer (47/1047/1049) … only report whether the alternate buffer is set"; mouse encodings/protocols are mutually exclusive and tracked separately]

**The failure chain:**
- A TUI (vim, `claude --rc`) enters alt-screen and enables mouse reporting (`\x1b[?1002h` / `?1003h` + `?1006h`). xterm now **routes the scroll wheel to the PTY as SGR mouse-event escape sequences** (`\x1b[<…M`) instead of scrolling its own buffer — this is correct, expected behavior *while the app is running*.
- The app is **killed/exits abnormally** (or restarted). The TUI never gets to send its own mouse-disable (`\x1b[?1002l` etc.) or `rmcup`.
- On the **restart path** the app writes only `\x1b[?1049l` — alt-screen exits, **but `term.modes.mouseTrackingMode` is still `'drag'`/`'any'`**. Now in the fresh shell, scroll-wheel events are *still* encoded as mouse-event bytes and sent to the shell, which echoes them as the `[%30/]`-style garble (the `<` / digits / `M` of an SGR mouse report mangled by the shell). Scroll-wheel no longer scrolls xterm's buffer.
- On the **abnormal-exit path** `term.reset()` *does* clear mouse modes (RIS), which is why the bug may appear intermittently — it manifests on the **restart** path (1049l only) and persists if a session was restarted while mouse mode was hot, but self-heals on a full crash that hits `reset()`. (This explains a "sometimes broken" report.)

### The fix (precise)

Re-derive SEAM B so **every** death/restart of a process explicitly disables mouse tracking, independent of alt-screen:

```ts
// Disable every mouse-tracking + encoding mode a TUI might have left on, THEN exit alt-screen.
// Order: disable mouse first so no stray wheel report races the buffer restore.
const MOUSE_RESET =
  '\x1b[?1000l\x1b[?1001l\x1b[?1002l\x1b[?1003l' + // X10 / highlight / button-event / any-event tracking
  '\x1b[?1004l' +                                  // focus reporting
  '\x1b[?1005l\x1b[?1006l\x1b[?1015l';             // UTF-8 / SGR / urxvt mouse encodings
const ALT_SCREEN_EXIT = '\x1b[?1049l';

// RESTART seam (D-07): preserve scrollback — NO term.reset().
term.write(MOUSE_RESET + ALT_SCREEN_EXIT);
term.write(`\r\n\x1b[2m— restarted ${hhmm} —\x1b[0m\r\n`);

// ABNORMAL-EXIT seam (D-05/D-13): the dead frame must not survive. Two valid options —
//   (a) MOUSE_RESET + ALT_SCREEN_EXIT (preserves scrollback, matches restart), OR
//   (b) term.reset() (RIS, clears modes AND wipes scrollback).
// Prefer (a) for consistency UNLESS the planner wants a guaranteed-blank frame on crash.
```

**Verification handle:** `term.modes.mouseTrackingMode` is a read-only getter (`'none' | 'x10' | 'vt200' | 'drag' | 'any'`). After the reset write, an assertion can confirm it is `'none'`. [VERIFIED: xterm.d.ts `IModes.mouseTrackingMode`, lines 1882–1890]. A smoke test can drive `\x1b[?1002h` then restart and assert `mouseTrackingMode === 'none'` and that a wheel event scrolls the buffer (`onScroll` fires / `viewportY` decreases) rather than producing PTY output.

**Discretion note:** If the planner cannot fully confirm via assertion, the debug plan is: (1) reproduce by running vim then header-Restart; (2) read `window.__sessionTerms[id].modes.mouseTrackingMode` before/after the 1049l write — it will read non-`none` after, confirming the cause; (3) apply MOUSE_RESET and re-read → `none`.

### The ephemeral gate — recommended shape

**"Ephemeral" is derivable, not a new stored boolean — but a stored flag is cleaner for the persistence filter.** Two viable designs (planner picks):

**Option A (derived predicate — minimal, no schema bump):** A session is *configured* iff it has any user-set metadata. `+ New` (`addSession`) creates a record with a default `name: "Session N"` and a default `icon: 🖥️` — so "has a name" is NOT a reliable signal. Add a `configured` predicate keyed on **explicit user intent**: any `updateProfile` call (name/icon/cwd/shell/startupCommand edited) flips it. Persist only configured records:

```ts
// store-schema.ts — keep version 1; add a pure predicate (unit-testable, no I/O)
export function isConfigured(rec: SessionRecord): boolean {
  return rec.configured === true; // explicit flag set by updateProfile (see Option B)
}
```
The problem: with a *derived* predicate you must distinguish "default name from `+ New`" vs "user typed a name." That is fragile. **Recommend Option B.**

**Option B (explicit `configured` boolean — RECOMMENDED, schema version bump):** Add one field to `SessionRecord`:

```ts
// src/shared/types.ts — SessionRecord gains:
  /**
   * True once the user has set ANY metadata (name/folder/cmd/icon) via the edit form
   * (D-02). Ephemeral (configured=false/undefined) sessions are NEVER persisted and
   * vanish on Close/exit; a configured session persists to the Inactive List.
   * Set true in PtyManager.updateProfile(); never reset to false (auto-promotion is
   * one-way). Default-undefined = ephemeral (a bare `+ New` shell).
   */
  configured?: boolean;
```
Wiring (all main-side; no bridge change — `updateProfile` is an existing channel):
1. `create()` builds records with `configured` absent → ephemeral. [pty-manager.ts 327–341]
2. `updateProfile()` sets `target.configured = true` whenever ANY field is written (the function already mutates name/cwd/shell/startupCommand/icon at lines 807–832 — add `target.configured = true;` before `signalStore()`). [VERIFIED: pty-manager.ts 791–834]
3. The store persists only configured records — filter at the snapshot boundary:
   ```ts
   // where index.ts does store.setSessions(ptyManager.listSessions()):
   store.setSessions(ptyManager.listSessions().filter((r) => r.configured === true));
   ```
   Equivalently add a `listConfiguredSessions()` to PtyManager so the filter is unit-tested in main.
4. `coerceOnLoad()` is unchanged — a persisted record was configured by definition; it stays `configured: true` on load (the coerce only touches `status`/`ptyPid`). Bump `SCHEMA_VERSION` to `2`; absent `configured` on a v1 record migrates to `true` (anything persisted under v1 was kept on purpose → treat as configured). [store-schema.ts 17, 42–44]

### D-05 exit→Inactive vs spawn-failure→error-card (transition wiring, main + renderer)
- **Configured exit on its own** (`exit`/Ctrl-D/crash, `userStopped===false`, `deriveStatus` → `exited`/`error`-but-was-running): the record should leave the live `sessions` map and re-enter `dormantRecords` so it appears in the Inactive List. **Currently** `onExit` keeps the record in `sessions` with `alive=false` (lines 479–484). The planner adds: on a self-exit of a *configured* session, move the record from `sessions` → `dormantRecords` (coerce status to `not_started`) so `listSessions()` renders it as a dormant Inactive-List entry. An **ephemeral** self-exit → `sessions.delete(id)` (gone). A **spawn-failure** (`pid -1`, the synchronous SC2 path at lines 276–284/306–316) already returns before the `sessions.set` — it stays an `error` row with a notice → the renderer keeps it in the Working Area as the error card (D-05). The distinguishing signal is *whether the session ever reached `running`*: a fork-then-die that reached `running` then crashed → Inactive (configured) / gone (ephemeral); a never-ran spawn-failure → error card stays.
- Renderer: `SessionManager` partitions on `status === 'not_started'` (dormant → Inactive List) vs live; the error row (`status === 'error'`, `activeIsCard` at line 500) renders the IdleCard error branch IN the Working Area (D-05). ## Axis 1 — Restart/scrollback bug (D-07): root cause + fix

### Current restart path (read this session)
- `SessionManager.handleRestart` → `window.api.ptyRestart(id)` → main `restart()` stops then respawns under the same `logicalId` (new `ptyPid`). [pty-manager.ts 650–690]
- The SAME `SessionView`/`Terminal` instance persists (keep-alive xterm, keyed on `id`), so scrollback *should* survive.
- On the second `running` status, `SessionView` writes `\x1b[?1049l` then the `— restarted HH:MM —` separator. [SessionView.tsx 306–318]

### Why repeated Restart trims scrollback (root cause — MEDIUM-HIGH confidence)
The trim is **not** in the separator write. The suspects, in order of likelihood:
1. **`term.reset()` firing on the intervening exit.** A restart goes through `stop()` → `onExit` with `userStopped=true` → status `stopped`. The abnormal-exit branch only resets on `exited`/`error` (line 331), so a clean user-stopped restart should NOT hit `reset()`. **BUT** if `stop()`'s SIGTERM→SIGKILL path produces a non-zero exit that races `deriveStatus` before `userStopped` is read, a restart could mis-classify as `error` → `term.reset()` → scrollback wiped. Verify `userStopped` is set true (line 631) *before* the kill, which it is — so this is a low-probability race, not the main cause.
2. **The `\x1b[?1049l` + alt-screen toggling discards primary rows.** Each restart that exits a never-entered alt-screen still pushes the alt-buffer enter/exit through xterm's buffer machinery; repeated toggling can trim the primary scrollback if the prior process left the buffer in the alt state. This is the most likely culprit for a *cumulative* (per-restart) trim.
3. **A respawned shell clearing its screen** (`clear` on login, or the readiness-probe output) consuming scrollback rows — benign but visible as "trim."

### Fix (D-07)
- **Guarantee restart NEVER calls `term.reset()`** — confirm the restart precursor is always `stopped` (userStopped), never `exited`/`error`. The mouse-mode-safe SEAM B fix above already removes `reset()` from the restart path entirely.
- **Only emit `\x1b[?1049l` when actually in the alt-screen.** Gate the write on `term.buffer.active.type === 'alternate'` (xterm exposes `buffer.active.type: 'normal' | 'alternate'`) so a restart of a plain shell doesn't toggle the alt-buffer needlessly. [VERIFIED: xterm.d.ts `IBuffer.type`]
- **Acceptance:** restart N times in a plain shell; assert the prior scrollback marker survives all N (the existing `alt-screen-reset.smoke.test.ts` only restarts twice — extend to ≥3 and assert the `KEEP_` marker persists every time).

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Reading the rendered terminal frame | A custom ANSI parser over the byte stream | `term.buffer.active.getLine(y).translateToString(true)` | xterm already maintains the interpreted grid; `record.cjs` proves it. |

The bug is that the app wasn't *reading* the mouse-mode state and wasn't resetting it. Detection and the D-13 fix are both "ask xterm what it already knows," not "re-derive from bytes."

## Common Pitfalls (Landmines)

### Pitfall 1: The `❯` trap (002 kill-finding)
**What goes wrong:** Treating `❯` as a waiting signal false-fires on Claude Code's persistent input caret (10/11 settles in the 002 capture).
**How to avoid:** `❯` is recorded for forensics but ABSENT from the decision. The real discriminators are `claude_footer` + `numbered_menu`. The current `agent-state.ts` `PROMPT_RE` (line 49) STILL includes `❯` — this is the exact line to fix.

### Pitfall 2: Stale-viewport false positive (001 finding #2)
**What goes wrong:** Scanning broad rows (or scrollback) matches a scrolled-up old menu above the active shell prompt → spurious WAITING.
**How to avoid:** Window the last ~4 *non-empty* lines (`record.cjs` `region`), and treat a shell prompt on the active line as authoritative FREE. Read only the viewport, never scrollback.

### Pitfall 3: Blank-padding window (001 finding #1)
**What goes wrong:** Classifying raw bottom rows when content sits at the top of the normal buffer → all-blank window → misclassify.
**How to avoid:** Filter to non-empty lines before windowing (`record.cjs` `nonEmpty`).

### Pitfall 4: Scrollback-trim from `term.reset()` (D-07)
**What goes wrong:** RIS (`term.reset()`) wipes scrollback; using it on the restart path trims history every restart.
**How to avoid:** Restart uses `\x1b[?1049l` (+ mouse reset), never RIS. Gate the 1049l on `buffer.active.type === 'alternate'`.

### Pitfall 5: Mouse mode left hot (D-13)
**What goes wrong:** `\x1b[?1049l` does NOT reset mouse tracking; scroll-wheel then garbles as `[%30/]` mouse-report bytes.
**How to avoid:** Always emit the full `MOUSE_RESET` disable on death/restart (see D-13 fix).

### Pitfall 6: Adding a bridge key
**What goes wrong:** `security.guard.test.ts` asserts exactly `EXPECTED_API_KEYS` (19). Any new key turns it RED.
**How to avoid:** Detection is renderer-local (no IPC). Ephemeral flag is main-internal (rides existing `updateProfile`). The Inactive List reuses `listSessions()`/`dormantRecords`. No new key this phase.

### Pitfall 7: Tearing down the keep-alive xterm on restart/hide
**What goes wrong:** Re-keying the mount effect on `active` or restart disposes the `Terminal` → scrollback lost, detector restarts.
**How to avoid:** Keep the mount effect keyed on `id` only (current behavior, line 391). The tick interval persists across in-place restart; just reset `lastHash`/`changeAt`.

### Pitfall 8: `— restarted —` on a fresh dormant Start (D-08)
**What goes wrong:** `hasRunBeforeRef` would be true for a restored session if it were promoted in the same SessionView instance — but a dormant restore mounts a FRESH SessionView (status `not_started` → no SessionView until Start), so `hasRunBeforeRef` starts false → no separator. **Confirm** the dormant→live Start does NOT reuse a SessionView that already had `hasRunBeforeRef===true`. The first Start of a restored session must show no separator.

## Open Questions
