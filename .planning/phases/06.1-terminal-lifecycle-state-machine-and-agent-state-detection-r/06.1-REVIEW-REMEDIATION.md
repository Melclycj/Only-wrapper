# Phase 06.1 — Code-Review Remediation (CR-01..CR-04 + WR-02)

Each finding was VERIFIED against the code (RED test where feasible) before any fix.
All five were judged REAL. Two (CR-01, CR-04) are defense-in-depth hardening rather
than active live bugs; both fixes are cheap, root-cause, and lock the invariant. No
human-approved behavior (amber / header Clear+Remove / Remove→Inactive / single Start)
was touched — only internals were hardened. No flags were flipped.

## Per-finding verdicts

### CR-01 — `quitting` flag never resets → flush skipped after a survived quit cycle — REAL, FIXED

**Verified:** `src/main/index.ts` `before-quit` gates the store flush on
`!quitting && store.isDirty()`. `quitting` is set `true` on the first pass and never
reset. The standard macOS `Cmd+Q` path exits the app (so the stale flag is moot), but
the flag is stuck `true` whenever a quit is *survived* — another `before-quit` listener
or an `onbeforeunload` cancels `app.quit()`, or a dock-relaunch follows a prevented
quit. In that state the next `before-quit` on the new window short-circuits to teardown
**without flushing**, so sessions created in the new window cycle are dropped on quit.
This is in the persistence path the round-3 work just fixed, so it is in-scope.

**Fix:** reset `quitting = false` inside `app.on('activate')` (the new-window-cycle
entry point), re-arming the flush-then-quit round-trip. This matches the review's
recommended fix exactly.

**Test note:** `index.ts` imports `electron` at module scope and has no unit harness
(consistent with the existing repo — there is no `index.test.ts`). The fix is a single
guarded assignment in the documented re-entrancy pattern; the durable-flush contract it
protects is already locked by `pty-recipe-persistence.test.ts` (the real `handleWindowClosed`
flush path) and `persistence.smoke.test.ts` (packaged-app round-trip). Adding an
index.ts unit test would require extracting the whole app-lifecycle wiring out of the
electron-coupled module — out of proportion to a one-line guard reset and outside this
remediation's scope. Documented here as a deliberate test-seam gap.

### CR-02 — concurrent `restart()` double-spawn (no in-flight lock) — REAL, FIXED

**Verified (RED):** `restart()` had no per-session lock. Two rapid `ptyRestart(id)`
calls each found the same live `s`, each armed an `onExit`→`respawn` listener on the
same pty, and the single exit fired BOTH → two `create()` calls (double startup-command
injection + an orphaned live pty). The per-closure `respawned` guard only protects one
listener, not two callers. RED test `restart() rejects/coalesces a concurrent second
call so only ONE respawn fires (CR-02)` failed with 3 spawned children (expected 2).

**Fix:** added `restarting: boolean` + `restartPromise?` to `PtySession`. A second
concurrent `restart()` returns the in-flight `restartPromise` (coalesce) instead of
arming a second listener. The lock lives on the OLD captured session; `create()` writes
a fresh `restarting:false` session over the id, naturally releasing it so a later
restart works (locked by the follow-on test).

### CR-03 — non-atomic dirty-clear in `flush()` → silent data loss — REAL, FIXED

**Verified (RED):** `flush()` set `this.dirty = false` BEFORE `await this.db.write()`.
A write rejection (disk full / EACCES) then left the store CLEAN with the mutation
unwritten, and `isDirty()` never re-armed the before-quit guard. A concurrent flush
(the scheduleSave timer racing before-quit) saw `dirty === false` and returned a no-op
while the first write was in flight — dropping the write if the first failed. Two RED
tests: re-arm-on-failure (got `isDirty()===false`, want `true`) and concurrent
coalescing.

**Fix:** clear `dirty` ONLY after `db.write()` resolves; re-arm `dirty = true` + rethrow
on failure; coalesce concurrent callers onto one in-flight `this.writing` promise. A
pre-existing debounce test (`debounces scheduleSave() bursts…`) asserted `isDirty()`
synchronously right after the trailing timer; under the corrected (post-write) clear it
now awaits the in-flight write the spy returns before asserting clean — the assertion was
relying on the old buggy synchronous clear, so updating it to await is correct, not a
relaxation.

### CR-04 — unclamped `setOrder` order poisons `nextOrder()` — REAL (defense-in-depth), FIXED

**Verified (RED):** `setOrder()` accepted any `Number.isFinite(order)` including
`Number.MAX_SAFE_INTEGER`. That poisons `nextOrder()` (`max + 1` loses precision at
MAX_SAFE_INTEGER), collapsing all subsequent new-session orders onto one colliding
value. RED test got `9007199254740991` (want ≤ 10000) and `3.9` (want floored to `3`).
Note: the review's `__proto__` Map-poisoning concern is a **false alarm** — JS `Map` is
prototype-pollution-safe (the review itself concedes this); only the `order`-clamp half
of CR-04 is a real defect, and that is what was fixed.

**Fix:** added `clampOrder(n) → 0..MAX_ORDER` (10000), mirroring the existing
`clampDimension` 1..1000 pattern, applied in `setOrder()` before writing to live or
dormant records. The renderer is not adversarial; this matches the established
validate-in-main defense-in-depth hardening.

### WR-02 — `handleRestart` optimistic `running` write lacks the `pid>0` guard — REAL, FIXED

**Verified:** `handleRestart` wrote `{ status: 'running' }` unconditionally.
`ptyRestart` → `restart()` → `create()` returns `{ pid: -1 }` on a failed respawn
(bad cwd / synchronous spawn throw) and broadcasts `error`; the unconditional optimistic
write can land after the error broadcast and clobber the error card. `handleStart`/
`handleStartNoCmd` already guarded with `pid > 0`.

**Fix:** extracted the optimistic flip into a pure `optimisticRunningFlip(rows, id, pid)`
helper (React/xterm-free, node-testable) that is a no-op when `pid <= 0` and clears any
stale `errorMessage` on success. `handleRestart`, `handleStart`, and `handleStartNoCmd`
now all route through it — one source of truth for the guard. Locked by
`optimistic-spawn.test.ts` (5 cases incl. the pid=-1 no-clobber and pid=0 cases).
