# Phase 06.1: Code Review Report

**Reviewed:** 2026-06-09
**Depth:** standard
**Files Reviewed:** 17
**Status:** issues_found

## Summary

Phase 06.1 implements a terminal lifecycle state machine (5 statuses + dormant/live two-bucket routing) plus a renderer-side agent-state detector (frame-stability + settle-independent fast path for the Claude permission screen). The architecture is well-decomposed — pure-function reducers extracted for testability, IPC boundaries respected, no electron imports in renderer, node-pty confined to main.

The most serious defects are: a race window where the `before-quit` handler disposes all PTYs but skips `unregisterIpc()` when the store is not dirty (IPC handlers dangle from a dead manager), a `restart()` double-spawn exposure on already-dead sessions that bypasses the `respawned` guard, silent data loss when `flush()` is concurrently called (double-clear of `dirty` flag), and a prototype-pollution vector in `setOrder()` via an unguarded `id as LogicalId` cast on attacker-controlled IPC input.

## Critical Issues

### CR-01: `before-quit` skips `unregisterIpc()` when store is not dirty — IPC handlers dangle

**File:** `src/main/index.ts:180-188`

**Issue:** The `before-quit` handler calls `ptyManager.disposeAll()` and `ptyManager.unregisterIpc()` only on the code path where `quitting` is already `true` (the second pass after the flush). But the second pass enters when `store.isDirty()` is `false` — meaning a quit that arrives while the store is already clean (e.g. the user has not done anything since the last save) goes directly to the teardown `else` branch, which DOES call both `disposeAll()` and `unregisterIpc()`. That path looks correct.

However, the first pass (`!quitting && store.isDirty()`) sets `quitting = true`, calls `event.preventDefault()`, fires the flush, then calls `app.quit()` — which re-fires `before-quit`. On that second fire `quitting` is `true`, so it falls through to `disposeAll()` and `unregisterIpc()`. This logic also appears correct.

The real defect is the **missing quitting-reset and double-quit guard on the `activate` path**. On macOS the dock-relaunch calls `createWindow()`, which calls `ptyManager.registerIpc(win)`. Because `ipcRegistered` was set `false` by `unregisterIpc()`, the handlers are re-registered correctly. But `quitting` is still `true` from the prior quit cycle, so the very next `before-quit` on the new window immediately skips the flush guard and goes straight to `disposeAll()` + `unregisterIpc()` — even if the store is dirty from work in the new window. Sessions created after a macOS dock-relaunch are never persisted.

**Fix:**
```typescript
app.on('activate', () => {
  if (BrowserWindow.getAllWindows().length === 0) {
    quitting = false; // reset the quit guard for the new window cycle
    createWindow();
  }
});
```

### CR-02: `restart()` on an already-dead session bypasses the `respawned` guard — potential double-spawn

**File:** `src/main/pty-manager.ts:794-812`

**Issue:** When `restart()` is called and `s.alive` is `false` (the session exited or errored before the restart request arrived), the `else` branch calls `respawn()` directly and synchronously. The `respawned` guard variable and the `off` disposable are declared inside the `if (s.alive)` block and never exist in the `else` path, which is fine for that case.

The real defect is that when `s.alive` is `true` at the time `restart()` is called, but the child exits TWICE before the disposable `off` fires (e.g. SIGTERM raises an exit, then the SIGKILL grace timer raises a second `onExit` delivery on some platforms or node-pty versions), the `respawned` flag correctly prevents a second `respawn()`. That part is guarded.

However, if `restart()` is called a second time while the first restart's stop-and-await is still in flight (the renderer dispatches two rapid `ptyRestart` calls), the second call finds the SAME `s` in `this.sessions` (still alive, or already not alive but overwritten by `create()` from the first restart's `respawn()`). If the first `respawn()` has already run and written a new `PtySession` to `this.sessions.get(id)`, the second `restart()` call reads that NEW session's `s.pty` and arms a fresh `off` listener on it, causing another stop+respawn. There is no in-flight restart lock guarding `this.sessions.get(id)`.

The comment at line 780 mentions "Capture the disposable and guard with `respawned`" but that guard only protects the single `onExit` listener — it does not prevent two concurrent `restart()` invocations from each arming their own listener on the live pty.

**Fix:** Add a per-session `restarting` flag (analogous to `removing`) that prevents a second concurrent `restart()` call from proceeding:
```typescript
// In PtySession:
restarting: boolean;

// In restart():
if (s.restarting) return Promise.reject(new Error(`restart: already restarting ${id}`));
s.restarting = true;
// ... in respawn():
s.restarting = false; // or on the new PtySession
```

### CR-03: `flush()` has a non-atomic dirty-flag clear — concurrent calls drop the write silently

**File:** `src/main/session-store.ts:214-222`

**Issue:** `flush()` clears the timer and dirty flag at the top, then awaits `this.db.write()`. If `scheduleSave()` is called between the `this.dirty = false` line (215) and the `await this.db.write()` completing (221), the new data mutation is marked dirty, but `flush()` is already in progress — when it resolves, the just-written data on disk may be stale (it captured `db.data` at the moment `write()` was called, but `db.data` is a live reference so lowdb writes the latest state regardless). However, there is a subtler problem: the `dirty = false` line clears the flag *before* the write resolves. If `flush()` is called concurrently (the `scheduleSave()` timer and the `before-quit` handler both call `flush()` at the same moment), the second call sees `dirty === false` and returns early (no-op path at line 219), while the first call is still awaiting `db.write()`. The second call returns without writing, and the `isDirty()` gate in `before-quit` will return `false` on the next check. If the first `db.write()` then fails (e.g. disk full), the data is silently lost — the `dirty` flag was already cleared and `isDirty()` will never re-arm the quit guard.

**Fix:** Move `this.dirty = false` to after the `await this.db.write()` resolves, and protect against concurrent re-entry with a promise chain:
```typescript
async flush(): Promise<void> {
  if (this.saveTimer) {
    clearTimeout(this.saveTimer);
    this.saveTimer = null;
  }
  if (!this.dirty || !this.db) return;
  // Clear dirty AFTER the write succeeds, not before.
  const db = this.db;
  this.dirty = false;
  try {
    await db.write();
  } catch (err) {
    this.dirty = true; // re-arm so the next flush retries
    throw err;
  }
}
```

### CR-04: `setOrder()` casts unchecked IPC string directly to `LogicalId` — prototype-pollution / map-poisoning

**File:** `src/main/pty-manager.ts:1056-1062`

**Issue:** The `setOrder()` handler receives a raw IPC payload. It type-guards each entry's `id` field as `typeof id !== 'string'` before the cast `id as LogicalId`. However, the validation does not check whether the string is actually a known `LogicalId` before passing it to `this.sessions.get(id as LogicalId)` and `this.dormantRecords.get(id as LogicalId)`. A renderer-side script (or a compromised renderer) can send `{ id: "__proto__", order: 0 }` — `typeof "__proto__" === 'string'` passes the guard, and the Map lookup `sessions.get("__proto__")` returns `undefined` on a real `Map` (Maps are prototype-pollution safe), so direct prototype pollution of the Map itself is not possible.

The real concern is `order` clamping: the `order` field is accepted as any `Number.isFinite(order)` value including `NaN`-equivalent edge cases are caught, but very large finite numbers (e.g. `Number.MAX_SAFE_INTEGER`) are accepted and written into `record.order`. This poisons the `nextOrder()` calculation (line 671-680): `max` becomes `Number.MAX_SAFE_INTEGER`, and the next new session gets `order = Number.MAX_SAFE_INTEGER + 1 = Number.MAX_SAFE_INTEGER` (loss of precision), and subsequent sessions all collide on the same order value. Because the IPC channel is renderer-controlled (a malicious or buggy renderer can send arbitrary JSON), this can make the app permanently unable to place new sessions in a meaningful order.

**Fix:** Clamp `order` to a sane range (e.g. `0..10000`) in `setOrder()`, matching the same defense-in-depth pattern used for `cols`/`rows`:
```typescript
const orderClamped = Math.max(0, Math.min(10000, Math.floor(order)));
// use orderClamped instead of order
```

## Warnings

### WR-01: `onPtyStatus` subscription re-created on every `sessions` state change — listener leak risk

**File:** `src/renderer/SessionManager.tsx:433-457`

**Issue:** The `useEffect` that subscribes to `window.api.onPtyStatus` lists `sessions` in its dependency array. Every time any session's status changes (which calls `setSessions`), React tears down the entire set of per-session listeners and immediately re-subscribes. This creates a brief window per status event where no listeners are attached. A `pty:status` event arriving during the teardown+resubscribe gap is silently dropped — the status badge stalls until the next event.

More critically, each subscription cycle calls `window.api.onPtyStatus(s.logicalId, handler)` for every session in the list. The `offs` cleanup array is constructed from the current `sessions` length, but between the teardown and re-subscription, new sessions added by `onAdd` or removed by `confirmClose` within the same React batch may cause the cleanup to register the wrong number of `off` callbacks, leaving orphan listeners.

The idiomatic fix is to subscribe once per `logicalId` (stable identity), not once per render cycle:

**Fix:** Move the status subscription into the `addSession` flow (subscribe when a new row is added, unsubscribe on removal), or use a `useRef`-backed registry keyed by `logicalId` that is managed outside the `sessions`-dependent effect. The keyboard-switch subscription at line 475 already demonstrates the correct pattern (subscribes once with empty deps, reads current state through `sessionsRef`).

### WR-02: `handleRestart` optimistically sets `status: 'running'` before `onPtyStatus` confirms it

**File:** `src/renderer/SessionManager.tsx:204-215`

**Issue:** `handleRestart` immediately writes `{ ptyPid: pid, status: 'running' }` to the matching row upon receiving the restart promise result. But `ptyRestart` in main calls `restart()`, which calls `create()`, which calls `setStatus(id, 'running', { ptyPid })` AND broadcasts via IPC. The `onPtyStatus` subscription will also receive the `running` status from main's broadcast. The optimistic write and the subscription-driven write race. The subscription write goes through `applyStatusEvent`, which correctly handles the transition; the optimistic write bypasses `applyStatusEvent` entirely.

This race can produce an incorrect state if the restart's new PTY fails to spawn (main broadcasts `error` after the `create()` error path) but the optimistic `running` write lands after the `error` broadcast (because the broadcast arrives while the `ipcMain.handle` return-value promise is still in flight). The error card would be immediately overwritten by the optimistic write showing a running session that is actually errored.

**Fix:** Remove the optimistic write in `handleRestart` and let the `onPtyStatus` subscription be the sole state driver for the restart path, mirroring the `handleStart` pattern (line 247: "A failed spawn returns pid -1 ... so do NOT optimistically flip to 'running'"). At minimum, guard it:
```typescript
if (pid > 0) {
  setSessions((prev) =>
    prev.map((row) =>
      row.logicalId === id ? { ...row, ptyPid: pid, status: 'running' } : row,
    ),
  );
}
```

### WR-03: `agentTick` interval captures `agentRunning` by closure over a `let` in an outer scope — state reset on restart may not propagate correctly

**File:** `src/renderer/SessionView.tsx:288-322`

**Issue:** `agentRunning`, `lastAgent`, and `tickState` are declared as `let` variables in the outer closure of the mount effect (lines 288-290). The `agentTick` setInterval captures them by closure reference. When the `onPtyStatus` handler fires with a non-running status (lines 438-442), it resets `agentRunning = false` and `tickState = initAgentTickState(performance.now())`. The next tick reads the updated `agentRunning` correctly.

However, there is a subtle timing issue: `tickState` is reassigned to a fresh `AgentTickState` object (line 440: `tickState = initAgentTickState(performance.now())`). The `decideAgentTick` function takes `state` by reference and mutates it in place (`state.waitingStreak += 1`, etc., in `agent-tick.ts:99-111`). If the tick fires between the reassignment of `tickState` and the `agentRunning = false` assignment (which precedes it at line 439), the tick reads the new `tickState` but `agentRunning` is still `true` — the tick executes a classify on the stale frame and may emit a spurious `waiting` state for the dead session.

This is a single-tick window and unlikely in practice, but since the `agentTick` interval is 100ms and `onPtyStatus` is an async IPC event, the race is real.

**Fix:** Set `agentRunning = false` BEFORE resetting `tickState`, or wrap both assignments atomically. The current code already does this in the correct order (line 439 before 440) for the tick gate. The real risk is the inverse: `tickState` reset happens, then the tick fires and reads the new empty `tickState` with `agentRunning` still true. Ensure the gate is checked first inside the tick:
```typescript
const agentTick = setInterval(() => {
  if (!agentRunning) return; // checked first, before any state read
  const next = decideAgentTick(tickState, viewportLines(), performance.now());
  if (next !== null) emitAgent(next);
}, TICK_MS);
```
This is already the case (line 319-320), so the actual mitigation needed is to set `agentRunning = false` atomically with the `tickState` reset — they already happen sequentially in the synchronous handler body, which is fine. **Downgrade note:** this is a real but very narrow race, only exploitable if a single JavaScript microtask boundary separates the two assignments, which does not happen in the current synchronous handler body. Flag as Warning for awareness.

### WR-04: `sanitizeNotice` in main preserves TAB (0x09) but drops LF (0x0A) — inconsistency with renderer's sanitizer

**File:** `src/main/pty-manager.ts:137` and `src/renderer/SessionView.tsx:117-119`

**Issue:** The main-side `sanitizeNotice` (pty-manager.ts:133-138) strips `0x00-0x08`, `0x0A-0x1F`, `0x7F-0x9F` — explicitly preserving TAB (0x09) but stripping LF (0x0A). The renderer-side `sanitizeNotice` (SessionView.tsx:117-119) strips `0x00-0x1F` and `0x7F-0x9F` — which strips BOTH TAB and LF.

A notice string with a TAB character passes through main's sanitizer intact and is then written by main into the `pty:status` payload. The renderer receives it and in `SessionView.tsx:384` writes it into the terminal: `` `\r\n\x1b[2m— ${sanitizeNotice(p.notice)} —\x1b[0m\r\n` ``. The renderer's `sanitizeNotice` strips the TAB there, but the `applyStatusEvent` path (which stores the notice as `errorMessage` for the IdleCard) does NOT call `sanitizeNotice` — `applyStatusEvent.ts:63` does `return { ...row, errorMessage: event.notice }` directly. So a TAB character in a cwd error notice is stored in `errorMessage` and rendered verbatim in the IdleCard's `{errorMessage ?? ...}` span, where it becomes a visual misalignment rather than a security issue.

More concretely: the `errorMessage` displayed in `IdleCard.tsx:117` is never sanitized before React renders it. React's JSX rendering escapes HTML entities, so no XSS is possible, but a notice string containing `\t` (preserved by main's sanitizer) produces a visual rendering artifact in the error card.

**Fix:** Either align both sanitizers to the same character set (strip all control chars including TAB in both), or sanitize `event.notice` in `applyStatusEvent` before storing it as `errorMessage`. The second option is preferable as it is the single entry point for all notice→errorMessage writes.

### WR-05: `close()` does not clear the SIGKILL grace timer if the session is alive — double-kill and timer leak

**File:** `src/main/pty-manager.ts:865-881`

**Issue:** `close(id)` checks `session.killTimer` and calls `clearTimeout` (line 873) — but only when the session exists in `this.sessions`. If `stop(id)` was called first (arming the SIGTERM→SIGKILL timer), and then `close(id)` is called before the kill timer fires, `close()` correctly clears it.

However, the check at line 867 (`if (!session)`) handles the dormant case and returns early without clearing any timer. The dormant case has no live pty and no kill timer, so this is fine. But: `close()` is only called from the renderer's `pty:close` IPC handler, and `removeLive()` routes through `stop()`. There is a scenario where the user calls `pty:stop` (which sets `removing=true` and arms a SIGTERM timer), and then before the child exits or the SIGKILL fires, a DELETE operation from the Inactive List sends `pty:close`. By that point the session is still in `this.sessions` (alive or dying), so `close()` does find it and clears the killTimer. This path looks correct.

The real gap: `close()` calls `session.pty.kill()` unconditionally (line 875) without checking `session.alive`. If the session's process already exited (e.g. self-exit between the Delete operation and the IPC round-trip), the kill call on a dead `pty` throws a native error. The `try/catch` on line 874-878 catches this, so it is not a crash — but the catch is described as "Already-dead children throw on kill" which is the legitimate case. The `!session.alive` guard used elsewhere (e.g. `stop()` line 731, `write()` line 833) is conspicuously absent here. A defensive `if (session.alive)` guard before the kill is the minor fix.

### WR-06: `rehydrateProfiles()` is called on EVERY new session add (`onAdd`) — unnecessary per-add latency and potential state clobber

**File:** `src/renderer/SessionManager.tsx:395-411`

**Issue:** `onAdd()` calls `void rehydrateProfiles()` after every `ptyCreate`. `rehydrateProfiles()` calls `window.api.listSessions()` (a round-trip to main) and merges `cwd/shell/startupCommand/configured` from main's authoritative snapshot into every row. This is done to get the resolved `cwd` and `shell` defaults for the edit modal.

The defect: `rehydrateProfiles` is called with `void` (fire-and-forget) and merges into ALL rows, not just the newly-added one. If two adds happen in rapid succession (fast double-click), the first `rehydrateProfiles` promise may resolve AFTER the second `setSessions` for the second add has already run. The merge `prev.map(row => ...)` sees both rows but applies the first `listSessions()` snapshot, which may not contain the second row yet (it was added after the first `listSessions()` request was issued). The second row is passed through with `if (!truth) return row` — which is safe. But the first row's `configured` field may be overwritten with a stale value from the snapshot that predates the first `ptyCreate` response landing.

This is a latency + cosmetic issue (the edit modal may briefly show a stale cwd for the first session after a double-add), not data loss, but the fire-and-forget pattern with `setSessions` inside an async closure is a known React concurrent-mode hazard.

**Fix:** Scope `rehydrateProfiles` to fetch only the newly-added `id` (pass the id as a parameter) and update only that row, or debounce the call, or remove it from `onAdd` entirely and only call it when the edit modal opens.

### IN-01: `classify()` shell-prompt regex does not match PowerShell `PS C:\...>` prompts

**File:** `src/shared/agent-state.ts:76`

**Issue:** The shell-prompt pattern `/\w[^>]>\s*$/.test(last)` is intended to match prompts like `PS C:\Users\foo>` (PowerShell). The regex requires at least two characters before `>`: `\w` (a word char) followed by `[^>]` (a non-`>` char) followed by `>`. For a standard PS prompt `PS C:\Users\foo>`, the last two chars before `>` are `o` (`\w`) and `o` (`[^>]`), which matches — so PS prompts do pass. However, a PS prompt ending in a drive letter like `PS C:\>` ends in `C:\>`, where the sequence is `C` (`\w`), `\` (`[^>]`), `>` — which matches. A 1-character directory like `PS D:\>` with nothing before the backslash similarly matches.

The edge case is a PowerShell prompt that ends in just `> ` with a space, like a custom `$` prompt (`> ` style). The first pattern `[$%#]\s*$` already catches `$` and `%`, but `>` alone (`> `) is NOT caught by either regex because `[^>]>` requires something before the `>`. This leaves PowerShell's default `>` prompt potentially unclassified as `free`. In practice, a PowerShell default prompt includes `PS path>` which is long enough to match the `\w[^>]>` pattern, so this is low-severity. Flag for documentation/test coverage.

### IN-02: `hasRendererIdentity()` and `hasIdentity()` use different fallback-name constants — divergence risk

**File:** `src/renderer/session-status.ts:28-29` and `src/main/session-identity.ts:42-43`

**Issue:** Both modules define the same two constants independently:
- `session-status.ts`: `const AUTO_NAME_RE = /^Session \d+$/;` and `const FALLBACK_NAME = 'Session';`
- `session-identity.ts`: `const AUTO_NAME_RE = /^Session \d+$/;` and `const FALLBACK_NAME = 'Session';`

These are identical literals today, but they are not shared from a single source of truth. If the auto-name format changes in one place (e.g. to `"New Session 1"` or locale-prefixed), the other module diverges silently, causing renderer and main to disagree on whether a session has identity — leading to incorrect self-exit routing (renderer keeps it in Working Area; main moves it to Inactive List, or vice versa).

**Fix:** Extract `AUTO_NAME_RE` and `FALLBACK_NAME` into `src/shared/names.ts` (or into `src/shared/types.ts`) and import from there in both modules.

### IN-03: `console.log` calls in `pty-manager.ts` — production logging artifact

**File:** `src/main/pty-manager.ts:378, 496, 501`

**Issue:** Three `console.log` calls remain in production code: the spawn notification (line 378), the readiness-probe timeout (line 496), and the PTY exit (line 501). These are described as "lifecycle logging" and tagged with security comments explaining why they must not log sensitive data. They are benign from a secrets perspective but violate the project's coding style (see `rules/typescript/coding-style.md`: "No `console.log` statements in production code"). In a production desktop app these emit to the Electron main process's stdout, which is invisible to users but could confuse debugging.

**Fix:** Replace with a structured logger (e.g. `electron-log` or a thin wrapper over `process.stdout`) with a configurable log level, or gate on a `DEBUG` environment variable. At minimum, downgrade to `console.debug` to signal they are diagnostics rather than operational output.
